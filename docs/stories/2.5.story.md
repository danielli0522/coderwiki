# Story 2.5: Document Generation - LLM Integration

## 基本信息

**Story ID:** 2.5  
**Epic:** Epic-2 (Repository Management)  
**Story 名称:** Document Generation - LLM Integration  
**优先级:** 高  
**估计工时:** 20小时  
**状态:** Draft  
**负责人:** 全栈开发人员  

## 用户故事

**作为** 系统用户，  
**我想要** 系统能够使用大语言模型自动生成高质量的技术文档，  
**以便** 获得准确、结构化的代码分析和技术文档。

## 验收标准

- [ ] 系统能够配置多种LLM API（OpenAI、Anthropic等）
- [ ] 系统能够根据代码分析结果生成结构化的技术文档
- [ ] 系统能够处理LLM API调用失败的情况并提供重试机制
- [ ] 系统能够缓存LLM响应以避免重复调用
- [ ] 系统能够监控和管理LLM API使用成本
- [ ] 系统能够生成包含项目概述、架构、API文档等完整内容
- [ ] 系统能够处理不同技术栈的文档生成需求
- [ ] 系统能够提供文档生成的进度反馈

## Dev Notes

### Previous Story Insights
根据Sprint 1的故事1.6和Story 2.1，数据库结构已经设计完成，包含文档表(documents)和任务表(tasks)的基本结构。LLM集成需要扩展这些模型以支持文档生成状态和LLM响应管理。

### Data Models
基于已设计的数据库结构，需要扩展以下数据模型：

```python
# backend/app/models/llm_config.py
class LLMConfig(db.Model):
    __tablename__ = 'llm_configs'
    
    id = db.Column(db.Integer, primary_key=True)
    user_id = db.Column(db.Integer, db.ForeignKey('users.id'), nullable=False)
    provider = db.Column(db.String(50), nullable=False)  # openai, anthropic, etc.
    model_name = db.Column(db.String(100), nullable=False)
    api_key = db.Column(db.String(500), nullable=False)
    base_url = db.Column(db.String(500))
    max_tokens = db.Column(db.Integer, default=4000)
    temperature = db.Column(db.Float, default=0.7)
    is_active = db.Column(db.Boolean, default=True)
    created_at = db.Column(db.DateTime, default=datetime.utcnow)
    updated_at = db.Column(db.DateTime, default=datetime.utcnow, onupdate=datetime.utcnow)
    
    # 关系
    user = db.relationship('User', backref='llm_configs')

# backend/app/models/document.py (扩展)
class Document(db.Model):
    __tablename__ = 'documents'
    
    # 现有字段...
    
    # LLM相关字段
    llm_config_id = db.Column(db.Integer, db.ForeignKey('llm_configs.id'))
    prompt_tokens = db.Column(db.Integer)
    completion_tokens = db.Column(db.Integer)
    total_tokens = db.Column(db.Integer)
    cost_estimate = db.Column(db.Float)
    generation_time = db.Column(db.Float)
    
    # 关系
    llm_config = db.relationship('LLMConfig', backref='documents')
```

### API Specifications
需要实现以下API接口：

1. **POST /api/llm/configs** - 创建LLM配置
   - 请求体：{"provider": "openai", "model_name": "gpt-4", "api_key": "...", "max_tokens": 4000}
   - 响应：{"success": true, "config_id": 1, "message": "LLM configuration created"}

2. **GET /api/llm/configs** - 获取用户LLM配置列表
   - 响应：{"configs": [{"id": 1, "provider": "openai", "model_name": "gpt-4", "is_active": true}]}

3. **POST /api/repositories/{id}/generate-docs** - 触发文档生成
   - 请求体：{"llm_config_id": 1, "doc_type": "technical"}
   - 响应：{"success": true, "task_id": 1, "message": "Document generation started"}

4. **GET /api/tasks/{id}/status** - 获取文档生成状态
   - 响应：{"status": "processing", "progress": 50, "message": "Generating API documentation"}

## 风险和注意事项

### 技术风险
- **API稳定性:** LLM API可能不稳定或变更
- **成本控制:** 无限制的API调用可能导致高额费用
- **响应质量:** LLM生成的文档质量可能不一致
- **性能问题:** 大型项目的文档生成可能耗时较长

### 解决方案
- 实现多供应商支持和故障转移
- 添加严格的成本监控和配额限制
- 实现质量检查和人工修正机制
- 使用异步处理和进度反馈

### 业务风险
- **用户体验:** 文档生成过程可能影响用户体验
- **数据安全:** API密钥和代码数据的安全保护
- **维护成本:** LLM API的持续维护和更新

### 解决方案
- 实现后台任务和状态通知
- 使用加密存储和访问控制
- 设计可扩展的架构支持多供应商
