### Implementation Details

#### 1. Backend Service (`/backend/app/services/mkdocs_service.py`)
- **Class**: `MkDocsService` 
- **Key Methods**:
  - `build_site_for_repository()` - Main site building logic
  - `get_site_status()` - Site status and metadata
  - `rebuild_site()` - Force rebuild existing site
  - `delete_site()` - Clean removal of site files
  - `list_user_sites()` - User site management

#### 2. API Endpoints (`/backend/app/api/mkdocs.py`)
- **Blueprint**: `mkdocs_bp` with `/api/mkdocs` prefix
- **Authentication**: All endpoints require `@login_required`
- **Error Handling**: Comprehensive exception handling with logging
- **Response Format**: Consistent JSON response structure

#### 3. Static File Serving (`/backend/app/routes/main.py`)
- **Route**: `/sites/<path:site_path>`
- **Security**: Path traversal prevention and user permission checks
- **Performance**: Proper MIME type handling and caching headers

#### 4. Frontend Integration (`/frontend/templates/repository/detail.html`)
- **Section**: MkDocs 静态文档站点 card in repository detail page
- **JavaScript Functions**:
  - `checkMkDocsSiteStatus()` - Status checking
  - `buildMkDocsSite()` - Site building with progress
  - `rebuildMkDocsSite()` - Rebuild with confirmation
  - `deleteMkDocsSite()` - Deletion with confirmation
  - `showMkDocsActions()` - Dynamic UI updates

### Site Generation Process

#### 1. Document Collection
```python
# From generated_docs directory and database
documents = self._collect_documents_for_repository(repository_id)
```

#### 2. Site Structure Creation
```
mkdocs-sites/repo_{id}_{name}/
├── docs/
│   ├── index.md           # Main documentation
│   ├── architecture.md    # Generated architecture docs  
│   ├── api.md            # API documentation
│   └── ...
├── mkdocs.yml            # Auto-generated configuration
└── site/                 # Built static files (after build)
    ├── index.html
    ├── assets/
    └── ...
```

#### 3. Configuration Generation
```yaml
site_name: "{Repository Name} Documentation"
theme:
  name: material
  features:
    - navigation.tabs
    - navigation.sections
    - navigation.expand
    - search.suggest
    - search.highlight
plugins:
  - search
  - awesome-pages
markdown_extensions:
  - admonition
  - codehilite
  - toc
  - tables
  - fenced_code
```

#### 4. Build Execution
```bash
cd {site_directory}
mkdocs build --clean --strict
```

### Performance Characteristics

- **Small docs (1-5 files)**: < 10 seconds build time
- **Medium docs (5-20 files)**: < 30 seconds build time  
- **Large docs (20+ files)**: < 60 seconds build time
- **Memory usage**: 50-200MB during build, 10-20MB at rest
- **Storage**: Built sites typically 2-3x source size + ~2MB theme

- `mkdocs-material==9.6.18` - Material Design theme
- `mkdocs-awesome-pages-plugin==2.10.1` - Page ordering plugin

## 风险和注意事项

### 技术风险
- **MkDocs依赖冲突**: MkDocs及其插件版本兼容性问题
- **构建超时**: 大型文档集可能导致构建超时
- **磁盘空间**: 大量站点可能占用过多存储空间
- **并发构建**: 多个用户同时构建可能影响性能

### 解决方案
- 严格控制包版本并定期测试兼容性
- 实现构建超时机制和进度监控
- 添加磁盘使用监控和清理策略
- 实现构建队列和资源限制

### 用户体验风险
- **构建等待时间**: 用户可能因构建时间过长而流失
- **操作复杂性**: 过多选项可能让用户困惑
- **错误恢复**: 构建失败后的恢复机制
- **站点访问**: 生成站点的发现和访问便利性

### 解决方案
- 提供清晰的进度指示和预估时间
- 保持界面简洁，提供合理的默认选项
- 实现自动重试和错误诊断
- 优化站点URL结构和导航方式

## 使用示例

### API使用示例
```python
import requests

# 登录获取会话
session = requests.Session()
session.post('http://localhost:5001/api/auth/login', json={
    'username': 'admin',
    'password': 'admin123'
})

# 构建站点
response = session.post('http://localhost:5001/api/mkdocs/repositories/1/build-site')
result = response.json()

if result['success']:
    print(f"站点URL: http://localhost:5001{result['site_url']}")
    print(f"生成文档数: {result['docs_count']}")
    print(f"构建时间: {result['build_time']}秒")

# 检查站点状态
status_response = session.get('http://localhost:5001/api/mkdocs/repositories/1/site-status')
status_data = status_response.json()
print(f"站点状态: {status_data}")
```
