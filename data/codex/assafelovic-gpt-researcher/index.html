01

从提问到报告：一次研究的完整旅程

想象你输入一个问题，几分钟后拿到一份带引用的研究报告——中间发生了什么？

GPT Researcher 是什么？

假设你是一位创业者，想知道「Nvidia 股价为什么一直在涨？」。你可以花一整天自己搜索、阅读、整理；也可以用 GPT Researcher——一个能自主完成整轮研究的 Agent。

你只需给它一个问题，它就会自动：

1
理解问题，拆解子问题

把大问题拆成多个搜索角度

2
全网搜索，抓取网页内容

并行搜索 20+ 来源，爬取网页正文

3
整理信息，生成研究报告

汇总去重，写出带引用的完整报告

💡
关键洞察

GPT Researcher 的灵感来自「计划-求解（Plan-and-Solve）」论文——先规划搜索策略，再逐个执行。这和人类做研究的方式一模一样：先列提纲，再填内容。

三行代码启动研究

GPT Researcher 的使用非常简单。你只需要创建一个 GPTResearcher 对象，传入你的问题，然后调用 conduct_research()：

CODE 
from gpt_researcher import GPTResearcher

query = "why is Nvidia stock going up?"
researcher = GPTResearcher(query=query)
research_result = await researcher.conduct_research()
report = await researcher.write_report()

认识核心角色：七大组件各司其职

就像一部电影有不同的演员，GPT Researcher 也有不同的组件扮演不同角色。

项目目录结构

先看看代码是怎么组织的。就像了解一个公司的部门设置一样，目录结构能帮你快速理解系统分工：

gpt_researcher/ 核心研究引擎
agent.py 主控类 GPTResearcher，系统的 CEO
actions/ 具体动作：搜索、爬虫、生成查询
skills/ 高级技能：研究执行、报告写作、深度研究
retrievers/ 20+ 搜索引擎适配器
scraper/ 网页爬虫工具箱
memory/ 向量存储和记忆管理
config/ 配置管理
multi_agents/ 多 Agent 协作模式（高级）

七个核心角色

GPT Researcher 内部有七个关键组件。把每个组件想象成一个团队成员：

🎯
GPTResearcher（主控）

团队 CEO。创建并协调所有其他组件。你只跟它打交道。

🔍
ResearchConductor（搜索指挥）

研究经理。规划搜索策略、生成子问题、协调搜索引擎并行工作。

📝
ReportGenerator（报告写手）

资深撰稿人。把收集到的信息整理成结构化的研究报告。

🧠
ContextManager（上下文管家）

资料管理员。把爬取到的内容做 向量嵌入，找到跟研究问题最相关的内容。

🕷️
BrowserManager（浏览器管家）

浏览器操作员。负责打开网页、提取正文内容、过滤广告和无关信息。

✂️
SourceCurator（素材筛选）

质量审核员。对搜索结果做排序和筛选，去掉重复和低质量来源。

🔬
DeepResearchSkill（深度研究）

进阶研究员。可选组件，用于更深入的多轮迭代研究。

🧩
设计模式：组合优于继承

GPT Researcher 没有把所有功能写在一个巨大的类里，而是把不同职责拆成独立的 Skill 类，每个都接收 self（主控的引用）。这叫「组合模式」——像搭积木一样组合功能。如果你想加新功能，只需要创建新的 Skill 类，不用改动主控代码。

关键洞察：上下文累积

注意 self.researcher.context 这个变量——它就像一个不断增长的文件夹。每一步搜索爬取的内容都会追加进去。最后报告生成器就是从这个「文件夹」里提取素材来写报告的。这种「逐步累积上下文」的模式在 AI Agent 中非常常见。

一个研究问题往往太大了，需要拆成几个角度来搜索。比如「Nvidia 为什么涨」可能被拆成「GPU 市场份额」「AI 芯片需求」「最新财报」等。看看代码怎么做：

实用提示：LLM 输出不稳定怎么办？

LLM 返回的 JSON 经常格式不规范。GPT Researcher 用 json_repair 库来修复——这是一个专门处理「LLM 返回的半合法 JSON」的工具。当你在构建 Agent 时，永远要考虑 LLM 输出的不稳定性。

网络爬虫与搜索：如何从全网抓取信息

GPT Researcher 的核心能力——搜索和爬虫——是如何实现的？

搜索引擎适配器：20+ 搜索引擎即插即用

GPT Researcher 支持超过 20 种搜索引擎。它用了一个巧妙的工厂模式——每种搜索引擎都是一个 Retriever 类，实现相同的接口。就像万能插座，不管你插什么插头都能用：

CODE 
# actions/retriever.py
def get_retriever(retriever: str):
    match retriever:
        case "google":
            from gpt_researcher.retrievers import GoogleSearch
            return GoogleSearch
        case "tavily":
            from gpt_researcher.retrievers import TavilySearch
            return TavilySearch
        case "duckduckgo":
            from gpt_researcher.retrievers import Duckduckgo
            return Duckduckgo
        case "arxiv":
            from gpt_researcher.retrievers import ArxivSearch
            return ArxivSearch
白话翻译

这是一个工厂函数：传入名字，返回对应的搜索引擎类

用 Python 3.10 的 match-case 语法（类似 switch）

如果用户要 Google 搜索...

导入 Google 搜索适配器

返回这个类（注意是类本身，不是实例）

如果要 Tavily 搜索...

导入 Tavily 适配器

如果要 DuckDuckGo...

如果要 ArXiv（学术论文搜索）...

以此类推，支持 20+ 搜索引擎

💡
设计模式：工厂模式 + 延迟导入

注意每个 case 分支都用 from ... import 做延迟导入——只有在真正使用某个搜索引擎时才加载它的代码。这样即使你只用了 Tavily，也不会因为没装 Google SDK 而报错。这是 Python 中非常实用的「按需加载」技巧。

爬虫系统：从 URL 到正文

搜索只是第一步——你拿到了一堆 URL，但 URL 里的内容才是你需要的。GPT Researcher 的 Scraper 系统负责从 URL 提取正文：

CODE 
# scraper/scraper.py
class Scraper:
    def __init__(self, urls, user_agent, scraper, worker_pool):
        unique_urls = list(dict.fromkeys(urls))
        self.urls = unique_urls

    async def run(self):
        contents = await asyncio.gather(
            *(self.extract_data_from_url(url, self.session)
              for url in self.urls)
        )
        res = [c for c in contents if c["raw_content"]]
        return res
白话翻译

爬虫类：接收 URL 列表，提取内容

初始化时去重——同一个 URL 不爬两次

用字典去重，保留原始顺序

核心方法：并行爬取所有 URL

asyncio.gather 让所有爬取同时进行（并行！）

对每个 URL 调用提取方法

过滤掉提取失败的（raw_content 为空的）

返回成功提取的内容列表

⚡
性能优化：asyncio.gather 并行爬取

asyncio.gather 是 Python 异步编程的核心技巧。它让所有 URL 的爬取同时进行，而不是一个一个排队。如果你要爬 20 个网页，每个花 2 秒，串行需要 40 秒，并行只需要 2 秒左右。这在网络密集型任务中是巨大的性能提升。

多种爬虫后端

不同网站需要不同的爬取策略。静态网页可以用简单的 HTML 解析，动态网页需要用浏览器模拟。GPT Researcher 支持多种后端：

BeautifulSoup 轻量级 HTML 解析器。适合静态网页，速度快，不需要浏览器
Browser (NoDriver) 无头浏览器爬虫。能执行 JavaScript，适合动态加载的网页
FireCrawl 专业爬虫 API 服务。质量高但需要付费 API Key
Tavily Extract Tavily 的内容提取 API。适合快速提取，集成简单
PyMuPDF PDF 文档解析器。专门处理 PDF 文件的提取
ArXiv Scraper 学术论文专用爬虫。提取 ArXiv 论文标题、摘要和正文

  • 实时性要求极高的新闻快讯
  • 纯中文环境下搜索引擎支持有限
  • 不需要报告、只需要简短回答的场景
  • 对 LLM 成本极度敏感的应用

源码中的硬限制

以下是直接从源码中提取的关键常量和限制：

max_iterations = 3 默认最多搜索 3 轮。可在配置中修改，但更多轮次意味着更高的 LLM API 成本
max_subtopics = 5 子主题报告最多 5 个子主题。这是 GPTResearcher.__init__ 中的默认值
temperature = 0.15 Agent 选择时的温度参数。低温度确保选择结果稳定可复现
20+ 搜索引擎 retrievers/ 目录下有 20+ 搜索适配器，每个都需要对应的 API Key
Python 3.11+ 最低 Python 版本要求。因为使用了 match-case 语法（3.10+）和类型注解新特性
MIT License 开源许可证，可自由商用。但部分搜索引擎 API 可能需要付费

成本与结论

1
LLM API 成本

每次研究至少调用 LLM 5-8 次（Agent 选择、子问题生成、每个子问题的搜索、报告生成）。使用 GPT-4o 级别模型，单次研究约 $0.05-0.15。

2
搜索引擎 API 成本

Tavily 有免费额度（1000 次/月），Google Custom Search 每天 100 次免费。DuckDuckGo 免费但稳定性较差。

3
时间成本

单次完整研究约 30-90 秒，取决于并行搜索数量和网页爬取速度。深度研究模式需要 3-5 分钟。

4
自托管 vs 云服务

代码完全开源可自托管。也可用 Docker 一键部署。前端支持轻量 HTML 版本和生产级 Next.js 版本。

🎯
一句话结论

GPT Researcher 是目前最成熟的开源研究 Agent 框架。如果你的项目需要「自动搜索+自动写报告」的能力，它是最快上手的选择。但要注意：它的输出质量受 LLM 和搜索引擎双重限制，不适合对事实准确性要求极高的场景。
