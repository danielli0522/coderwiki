核心方法：conduct_research()

整个数据管道的入口是 conduct_research()。看看它怎么协调搜索流程：

CODE 
# skills/researcher.py
async def conduct_research(self):
    # Step 1: Choose agent type
    self.researcher.agent, self.researcher.role = \
        await choose_agent(
            query=self.researcher.query,
            cfg=self.researcher.cfg)

    # Step 2: Search based on source type
    if self.researcher.report_source == ReportSource.Web.value:
        research_data = await \
            self._get_context_by_web_search(
                self.researcher.query, [],
                self.researcher.query_domains)

关键洞察：上下文累积

注意 self.researcher.context 这个变量——它就像一个不断增长的文件夹。每一步搜索爬取的内容都会追加进去。最后报告生成器就是从这个「文件夹」里提取素材来写报告的。这种「逐步累积上下文」的模式在 AI Agent 中非常常见。

子问题是怎么生成的？

一个研究问题往往太大了，需要拆成几个角度来搜索。比如「Nvidia 为什么涨」可能被拆成「GPU 市场份额」「AI 芯片需求」「最新财报」等。看看代码怎么做：

白话翻译

用 Prompt 模板生成一个「子问题生成指令」

传入：原始问题、父问题（如果有）、报告类型

最多搜索 3 轮

附带初始搜索结果作为上下文参考

把这个指令发给 LLM（大语言模型）

LLM 会返回一组子问题（JSON 格式）

用 json_repair 解析可能格式不规范的 JSON

返回子问题列表，比如 ["市场份额","AI需求","财报"]

⚠️
实用提示：LLM 输出不稳定怎么办？

LLM 返回的 JSON 经常格式不规范。GPT Researcher 用 json_repair 库来修复——这是一个专门处理「LLM 返回的半合法 JSON」的工具。当你在构建 Agent 时，永远要考虑 LLM 输出的不稳定性。
