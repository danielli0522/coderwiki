MCP Server 如何启动、连接和管理 Chrome 实例？两种模式的区别是什么？

两种连接模式

browser.ts 提供了两种浏览器连接方式，通过配置参数自动选择：

🚀

Launch 模式（默认）

MCP Server 自己启动一个 Chrome 实例。使用 pipe 通信，拥有完整控制权。支持 headless、isolated、自定义 viewport 等配置。

🔗

Connect 模式

连接到已运行的 Chrome 实例（通过 browserUrl、wsEndpoint 或 autoConnect）。适合共享状态、沙箱环境或远程调试场景。

ℹ
关键设计

Launch 模式关闭时会 browser.close()（杀掉子进程），Connect 模式关闭时只 browser.disconnect()（保留用户的 Chrome）。这确保了连接到用户日常浏览器时不会意外关掉它。

McpPage：单页状态封装

每个浏览器标签页都有一个对应的 McpPage 实例，它封装了：

TypeScript export class McpPage implements ContextPage { readonly pptrPage: Page; // Puppeteer Page 对象 readonly id: number; // 自增 ID，供工具引用 textSnapshot: TextSnapshot; // 当前页面的 a11y 树快照 emulationSettings: EmulationSettings; // 模拟配置 #dialog?: Dialog; // 未处理的浏览器对话框 devToolsPage?: Page; // 关联的 DevTools 窗口 // 关键方法 getElementByUid(uid): Promise<ElementHandle>; waitForEventsAfterAction(action): Promise<Result>; throwIfDialogOpen(): void; }
中文解读
McpPage 是单页面的状态容器，每个标签页一个实例。
pptrPage 持有 Puppeteer 的 Page 引用，id 是自增编号供工具调用时引用。
textSnapshot 存储无障碍树快照，工具通过 uid 定位 DOM 元素。
emulationSettings 保存网络节流、CPU 节流、地理位置等模拟配置。
dialog 字段跟踪弹窗状态，某些工具执行前必须先处理弹窗。
getElementByUid 将快照中的 uid 解析为真实的 DOM 元素句柄。
waitForEventsAfterAction 是核心等待机制，确保操作后 DOM 稳定再返回。
