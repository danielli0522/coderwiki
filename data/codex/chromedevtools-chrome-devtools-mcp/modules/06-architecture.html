Puppeteer vs Playwright：为什么选 Puppeteer

在浏览器自动化领域，Puppeteer 和 Playwright 是两个主要选择。Chrome DevTools MCP 选择 Puppeteer 的理由：

  • ✅ Google 官方维护：Chrome DevTools 团队自己开发，对 Chrome 的支持最及时
  • ✅ CDP 原生支持：直接通过 Chrome DevTools Protocol 与浏览器通信，支持 pipe 和 WebSocket
  • ✅ DevTools Frontend 集成：能直接使用 chrome-devtools-frontend 包的 Trace 解析和 HeapSnapshot 引擎
  • ✅ Extension 支持：Puppeteer 原生支持 Chrome 扩展的安装、触发和管理
  • ✅ DevTools 页面操作：handleDevToolsAsPage 选项让 DevTools 窗口也作为可操作页面
  • ⚠️ 只支持 Chrome：不支持 Firefox、Safari 等浏览器（项目定位就是 Chrome 专用）
  • ⚠️ API 稳定性：Puppeteer 的某些内部 API（如 _tabId）没有类型保证
  • ⚠️ DevTools Frontend 耦合：依赖 chrome-devtools-frontend npm 包的特定版本，升级成本高
  • ✅ 合理取舍：对于"让 AI 操控 Chrome 调试"这个场景，Puppeteer 是最佳选择

DevtoolsUtils.ts 的 UniverseManager

UniverseManager 为每个页面创建独立的 DevTools "Universe"（包含 CDP Session 和 Worker），使多个页面可以并行使用 DevTools 引擎而互不干扰。这是通过 PuppeteerDevToolsConnection 适配器实现的，它将 Puppeteer 的 CDPSession 包装成 DevTools 期望的 CDPConnection 接口。
