01

UI-TARS 是什么：让 AI 看懂屏幕

想象你对着电脑说"帮我在携程订一张北京到上海的机票"——AI 看着屏幕，找到按钮、填写表单、点击提交。这不是科幻，这就是 UI-TARS。

这个项目解决什么问题？

传统自动化需要写死每个操作步骤——"点击坐标(320, 480)"、"在 id 为 username 的框里输入文字"。一旦界面改版，脚本全废。UI-TARS 换了个思路：让 AI 看懂屏幕，像人一样理解和操作。

💡
核心洞察

UI-TARS 的哲学是：与其为每个应用写专门的 API 集成，不如让 AI 直接"看"屏幕，理解界面元素，像人一样操作。这样任何应用都能用，不需要逐个适配。

🖥️

Agent TARS

多模态 AI Agent 框架，支持 CLI 和 Web UI，能通过视觉能力操作电脑和浏览器

🎮

UI-TARS Desktop

Electron 桌面应用，提供本地和远程的电脑/浏览器 GUI 操作能力

🧩

Tarko 框架

底层引擎，提供 Agent 循环、事件流、MCP 工具调用、LLM 通信等核心能力

项目结构一览

02

GUI Agent 循环：截图 → 思考 → 操作

想象一个棋手：看棋盘、思考下一步、落子——然后重复。GUI Agent 就是 AI 版的棋手，只不过棋盘是你的电脑屏幕。

核心循环：看 → 想 → 做

GUIAgent 类实现了 UI-TARS 最核心的循环。每一步都是：截取屏幕截图，交给视觉语言模型分析，模型返回"下一步该做什么"，然后执行操作。

1
截图（Screenshot）

用 Electron 的 desktopCapturer 捕获屏幕图像，转为 base64 编码

2
推理（Inference）

多模态 LLM 理解截图内容，输出 "Thought: ... Action: ..." 格式的指令

3
解析（Parse）

action-parser 把文字指令解析成结构化操作：点击坐标、输入文字、按键组合等

4
执行（Execute）

Operator（NutJS / 浏览器 / ADB）执行实际操作，然后回到第 1 步继续循环

GUIAgent 的循环代码

这段代码是 GUIAgent.run() 的核心——一个 while(true) 循环，不断截图、推理、执行，直到任务完成或达到最大循环次数。

CODE 

// packages/ui-tars/sdk/src/GUIAgent.ts
while (true) {
  if (loopCnt >= maxLoopCount) {
    data.status = StatusEnum.ERROR;
    break;
  }
  loopCnt += 1;

  // Step 1: 截图
  const snapshot =
    await operator.screenshot();

  // Step 2: 模型推理
  const { prediction,
    parsedPredictions } =
    await model.invoke(vlmParams);

  // Step 3: 执行每个操作
  for (const p of parsedPredictions) {
    if (p.action_type === 'finished')
      break;
    await operator.execute({
      parsedPrediction: p });
  }
}
PLAIN ENGLISH

进入无限循环——直到任务完成才会退出...

安全阀：如果循环次数超过 maxLoopCount（默认 25），强制停止...

状态设为 ERROR，跳出循环...

循环计数器加 1...

第一步：调用 operator 截取当前屏幕...

第二步：把截图和历史对话发给视觉语言模型...

模型返回预测结果和解析后的操作列表...

第三步：逐个执行模型建议的操作...

如果模型说 finished，说明任务完成了，跳出循环...

否则执行操作（点击、输入、滚动等），然后回到循环开头继续截图。

Operator：从 AI 指令到鼠标点击

AI 说"点击(335, 360)"，但电脑怎么知道你要点哪里？Operator 就是 AI 世界和真实电脑之间的翻译官。

模型输出的是文字指令——"click(start_box='[0.131, 0.25, 0.131, 0.25]')"。但电脑需要真实的鼠标移动和键盘输入。Operator 就是这个翻译层。

Electron 桌面截图的实现

在 Electron 桌面应用中，截图不是用浏览器 API，而是用 Electron 原生的 desktopCapturer。这段代码展示了如何获取高质量屏幕截图。

CODE 

// apps/ui-tars/src/main/agent/operator.ts
export class NutJSElectronOperator
  extends NutJSOperator {

  public async screenshot() {
    const { physicalSize, logicalSize,
      scaleFactor } = getScreenSize();

    const sources = await
      desktopCapturer.getSources({
        types: ['screen'],
        thumbnailSize: {
          width: logicalSize.width,
          height: logicalSize.height }
      });

    const screenshot =
      sources[0].thumbnail;
    return {
      base64: screenshot.toJPEG(75)
        .toString('base64'),
      scaleFactor
    };
  }
}
PLAIN ENGLISH

继承 NutJSOperator，扩展 Electron 特有的截图能力...

获取屏幕的物理尺寸、逻辑尺寸和缩放比例...

调用 Electron 的 desktopCapturer API 获取屏幕截图...

只截取屏幕（不是窗口）...

按逻辑尺寸获取缩略图（考虑 Retina 缩放）...

取第一个屏幕源（主显示器）...

转为 JPEG 格式（质量 75%），再转 base64 文本...

返回 base64 图片和缩放因子——供模型使用。

MCP 是 AI 世界的"万能插座"——任何工具只要插上，AI 就能用。搜索引擎、文件操作、代码执行，统统变成 AI 的手和脚。

MCPAgent 的代码实现

MCPAgent 继承自 Tarko 的基础 Agent 类，在初始化时连接所有 MCP Server，把它们的工具注册到自己身上。

CODE 

// multimodal/tarko/mcp-agent/src/mcp-agent.ts
export class MCPAgent<T>
  extends Agent<T> {
  private mcpClients:
    Map<string, IMCPClient>;

  async initialize() {
    for (const [name, config]
      of Object.entries(mcpConfig)) {
      const client =
        new MCPClientV2(name, config);
      await client.initialize();

      const tools = client.getTools();
      for (const tool of tools) {
        this.registerTool(
          new Tool({ id: tool.name,
            function: (args) =>
              client.callTool(tool.name, args)
          })
        );
      }
    }
  }
}
PLAIN ENGLISH

MCPAgent 继承自基础 Agent 类，专门处理 MCP 工具调用...

用一个 Map 管理所有 MCP 客户端连接...

初始化时遍历配置中的所有 MCP Server...

取出每个 Server 的名称和连接配置...

创建 MCP 客户端并建立连接...

获取这个 Server 提供的所有工具...

逐个注册为 Agent 可用的工具...

工具的执行函数就是调用 MCP 客户端的 callTool 方法...

传入工具名和参数，等待结果返回。

🔑
设计模式：依赖注入

MCPAgent 不知道也不关心它调用的工具是怎么实现的。它只通过 MCP 协议和工具通信。你可以换掉底层的搜索引擎、换掉爬虫实现——Agent 的代码一行都不用改。这就是 依赖注入 的威力。

模型与 Prompt：AI 的眼睛和大脑

模型适配层：UITarsModel

UI-TARS 支持多个视觉语言模型。UITarsModel 封装了 OpenAI SDK，统一了不同模型的调用方式。

CODE 

// packages/ui-tars/sdk/src/Model.ts
export class UITarsModel extends Model {
  async invoke(params: InvokeParams) {
    const { conversations, images,
      screenContext, uiTarsVersion } = params;

    // 压缩图片：不超过模型像素上限
    const compressed = await
      Promise.all(images.map(img =>
        preprocessResizeImage(
          img, maxPixels)));

    // 转换为 OpenAI 消息格式
    const messages =
      convertToOpenAIMessages(
        { conversations, images: compressed });

    // 调用模型并解析操作
    const { prediction } =
      await this.invokeModelProvider(...);
    const { parsed } = actionParser(
      { prediction, screenContext });
    return { prediction, parsedPredictions: parsed };
  }
}
PLAIN ENGLISH

UITarsModel 负责和视觉语言模型通信...

接收对话历史、图片、屏幕尺寸和模型版本...

第一步：压缩图片，确保不超过模型的像素上限...

每张图片独立压缩，用 Promise.all 并行处理...

第二步：把对话和图片转成 OpenAI 的消息格式...

截图作为 image_url 消息，文本作为 text 消息...

第三步：调用实际的模型 API...

第四步：用 actionParser 解析模型输出为结构化操作...

返回原始预测文本和解析后的操作列表。

适合 vs 不适合

✓ 适合

  • ✓ 自动化操作没有 API 的桌面应用或网站
  • ✓ 界面经常变化、难以维护固定选择器的自动化
  • ✓ 需要跨平台（桌面+浏览器+手机）混合操作

  • ✓ 快速原型验证——用自然语言描述任务即可执行
  • ✓ 可接受 LLM API 调用成本的生产环境

✗ 不适合

  • ✗ 需要毫秒级精确操作的高频交易等场景
  • ✗ 大规模批量数据处理（效率不如专用脚本）
  • ✗ 预算有限无法承担 LLM API 费用
  • ✗ 需要 100% 确定性的回归测试（AI 有随机性）
  • ✗ 完全离线/无网络的环境
  • ✗ 涉及高度敏感操作（金融交易）无人监督的自动化
⚠️
成本估算

GUI Agent 每次循环（截图+模型推理）是一次多模态 API 调用。复杂任务需要 10-50 次循环。以 Claude 3.5 Sonnet 为例，单次截图约 1000 tokens 输入 + 200 tokens 输出，一个 30 步任务大约消耗 $0.1-0.5。生产使用前务必评估成本。

源码中的硬限制

MAX_LOOP_COUNT = 25单次任务最大循环次数，防止 Agent 无限运行（来源：GUIAgent.ts）
MAX_SNAPSHOT_ERR_CNT = 10连续截图失败上限（来源：constants.ts）
MAX_IMAGE_LENGTH = 5对话历史最大保留图片数，超出的老图片会被裁剪（来源：utils.ts）
maxIterations = 1000Tarko Agent 框架默认最大迭代次数（来源：agent.ts）
maxTokens: V1.0=1000, V1.5=65535不同模型版本的最大输出 token 数（来源：Model.ts）
Node.js >= 22Agent TARS CLI 要求 Node.js 22 或更高版本（来源：README.md）
