Building the Knowledge Map

How CodeGraph reads every file, extracts every symbol, and stores every connection

The X-Ray Machine for Code

When CodeGraph reads a source file, it doesn't just scan for text. It runs each file through a parser that produces an Abstract Syntax Tree (AST).

Think of it like an X-ray. A photograph of a person shows skin, clothing, and hair. An X-ray shows bones — the underlying structure that everything else rests on. The AST is the skeleton of your code: which lines are function declarations, which are method calls, which are import statements.

🦴
Why AST Instead of Text Search?

A text search for "login" finds every comment, string, and variable name containing those letters. An AST query finds exactly "function declarations named login" — nothing more. That precision is what makes CodeGraph's results reliable, not just approximate.

Tree-Sitter: The Universal Parser

CodeGraph uses tree-sitter — one engine that handles 19+ languages using separate grammar modules. Each grammar is compiled to WASM.

src/extraction/tree-sitter.ts (simplified) async function extractFromSource( source: string, language: Language ) { const parser = new Parser(); const grammar = await getLanguage(language); parser.setLanguage(grammar); const tree = parser.parse(source); // Query tree for functions, classes, imports... return extractNodes(tree, language); }
Plain English
Function that takes source code text + language name
What's the source file content?
Which language is it? (TypeScript, Python, Go…)
Create a parser engine
Load the grammar rules for this language (from WASM file)
Tell the parser which language to expect
Parse the text → produce an AST (a tree of nodes)
Find functions, classes, methods, imports in the tree
Return the list of nodes and edges found
🗂️
19+ Languages, One Engine

TypeScript, JavaScript, Python, Go, Rust, Java, C#, PHP, Ruby, C, C++, Swift, Kotlin, Dart, Svelte, Vue, Liquid, Pascal, Scala. Each has its own grammar module (.wasm file), loaded on demand. Adding a new language is as simple as adding a new grammar.

Reference Resolution: Following the Threads

After the AST pass, CodeGraph knows every symbol that exists — but doesn't yet know how they connect across files. That's the resolver's job: follow every import statement to its definition.

src/resolution/import-resolver.ts (simplified) function resolveViaImport( fromFile: string, importPath: string, projectRoot: string ) { let resolved = path.resolve( path.dirname(fromFile), importPath ); for (const ext of ['.ts', '.js', '.tsx']) { if (fs.existsSync(resolved + ext)) return resolved + ext; } return null; // unresolved }
Plain English
Function: "where does this import actually point?"
The file doing the importing
The import path written in code (e.g. './utils')
The project root (for absolute path resolution)
Compute the candidate file path by joining directory + import path
Try each common file extension (.ts, .js, .tsx)
If the file exists with this extension, return the full path
Couldn't find it — mark as unresolved for later retry
🔗
Four Resolution Strategies

CodeGraph tries four approaches for every unresolved reference: (1) Follow the import path, (2) Resolve TypeScript path aliases from tsconfig.json, (3) Fuzzy name-match across all known symbols, (4) Framework-specific route detection. If all four fail, the reference stays marked "unresolved."
