The Parallel Brain: Worker Threads

Parsing 25,000 files is slow. If CodeGraph did that work on the main thread, your terminal would freeze for minutes while the progress bar crept along. Instead, CodeGraph offloads all heavy parsing to a worker thread.

But there's a subtle problem: WASM heaps never shrink. Every file parsed consumes a little more heap memory — and that memory is never released. Parse enough files and the worker runs out of memory and crashes.

src/extraction/parse-worker.ts (key logic) const FILE_RECYCLE_LIMIT = 250; let filesProcessed = 0; async function processFile(filePath: string) { filesProcessed++; if (filesProcessed >= FILE_RECYCLE_LIMIT) { // WASM heap never shrinks — recycle the worker await restartWorker(); filesProcessed = 0; } return extractFromFile(filePath); }
Plain English
After 250 files, kill and restart the worker process
How many files this worker has parsed so far
Parse one file…
Count it
Have we hit 250 files?
Yes — the WASM heap is filling up
Start a fresh worker process (empty WASM heap)
Reset the counter
Actually parse this file and return the results
🧠
Why 250 Files Specifically?

This is based on empirical testing. At 250 files, the WASM heap is large enough to matter but small enough that recycling overhead is negligible. The fix is a workaround for a fundamental V8 engine limitation — not a CodeGraph bug, but a quirk of the runtime it runs on.

The Backup Generator: Native vs WASM SQLite

CodeGraph needs SQLite for its database. The fastest way to use SQLite in Node.js is via a native addon (better-sqlite3). But native addons require a C compiler to build — which fails in some Docker containers, CI environments, and Windows setups without build tools.

So CodeGraph uses an auto-detection pattern: try the fast native version first; if it fails, silently fall back to the pure-JavaScript WASM version. Same API, different speed.

src/db/sqlite-adapter.ts (simplified) async function createDatabase(path: string) { try { const { Database } = await import('better-sqlite3'); return new Database(path); // fast } catch { const { Database } = await import('node-sqlite3-wasm'); return new Database(path); // portable } }
Plain English
Create a database connection — figure out which engine to use
Try first…
Load the native C++ SQLite binding
Open the database using native code (~5x faster)
If that fails (no build tools, wrong platform)…
Load the pure-JavaScript WASM version instead
Open the database — slower but works everywhere
Caller gets a database — doesn't need to know which engine
🔋
The "Try Native, Fall Back to WASM" Pattern

This pattern — attempt the fast native path, silently fall back to portable — appears all over production Node.js libraries (sharp, canvas, argon2). It's how you ship software that works on a developer's MacBook, a teammate's Windows machine, and a Linux Docker container all at once.

The Ripple Calculator: Impact Radius BFS

One of CodeGraph's most powerful features is impact analysis: "If I change this function, what else breaks?" It uses a BFS traversal starting from the changed symbol, following edges outward to find everything in the dependency chain.

🕸️
getCallers(name)

BFS traversal following inbound "calls" edges. Who calls this function? All callers, up to maxDepth=3 levels up the call tree.

🌊
getCallees(name)

BFS traversal following outbound "calls" edges. What does this function call? All dependencies, down into the call tree.

💥
getImpactRadius(name)

Multi-level BFS in both directions. The full blast radius of a change — everything that could break if this symbol changes.

🎯
Why BFS and Not DFS?

DFS could explore infinitely deep call chains (recursive functions, circular references). BFS with a depth limit (maxDepth=3) naturally stops at a useful boundary — close enough to be relevant, not so far that you get every function in the codebase.
