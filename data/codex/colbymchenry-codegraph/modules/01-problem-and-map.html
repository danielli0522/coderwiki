AI 代理的困境与代码地图

当 Claude Code 需要理解你的代码库时，它在经历什么？

想象一下：你在陌生城市开车

没有地图，没有导航。想找到一家咖啡馆，你只能一条街一条街地开过去看。每经过一个路口，你就花掉一点时间和汽油。

这就是今天 AI 编程代理 面对代码库的处境——它用 grep 搜索、用 Read 翻文件、用 glob 找路径。每一次调用都要消耗 token 和时间。

💡
关键洞察

在没有地图的城市里，找一个地点要挨家挨户问。而有了地图索引，你一眼就知道目标在哪里、怎么走最近。CodeGraph 就是给 AI 代理装的那张地图。

没有地图的代价有多大？

CodeGraph 作者在 7 个真实开源项目上做了基准测试，让 Claude Code 回答同一个架构问题，对比有/无 CodeGraph 的情况：

💰

平均省 35% 成本

大项目（如 VS Code ~10k 文件）省 26%，Tokio（Rust）省 82%——代码库越大，省得越多。

📏

平均减 57% Token

Excalidraw 项目从 3.5M token 降到 344K——AI 不再反复读同样的文件。

⚡

平均快 46%

Excalidraw 回答时间从 2m58s 降到 48s。因为不需要反复 grep/read 来回折腾。

🔧

平均减 71% 工具调用

Excalidraw 从 79 次 Read/Grep 降到 3 次。一次图谱查询顶几十次文件翻阅。

没有地图时，AI 在做什么？

下面是 Claude Code 在没有 CodeGraph 时的典型行为模式——它会启动 Explore 子代理，反复搜索：

CODE 

// AI 代理的典型探索模式（无 CodeGraph）

grep("handleRequest")     // 第1次搜索：找函数在哪
read("src/server.ts")       // 第2步：读那个文件
grep("validateInput")      // 第3次搜索：找它调用了什么
read("src/validators.ts")   // 第4步：又读一个文件
grep("saveToDatabase")      // 第5次搜索：继续追调用链
// ... 如此反复，直到拼凑出完整路径
大白话

AI 需要找到 "handleRequest" 函数在哪——就像在一栋大楼里挨层找人。

找到了，打开那个文件看看里面写了什么。

发现它调用了 "validateInput"——继续找这个函数在哪。

又打开另一个文件。每一步都要消耗 token 和时间。

还要继续追 "saveToDatabase" 在哪里定义的。

整个过程就像没有地图的快递员，每送一单都要重新认路。

CodeGraph：预建好的代码地图

CodeGraph 在你写代码的时候就把整个代码库解析成了一张 知识图谱——哪个函数调了谁、哪个类继承了谁、文件之间的依赖关系，全部预先建好索引，存在本地的 SQLite 数据库里。
