核心角色 -- 30 个专业 Agent 如何分工

每个 Agent 就像一个科室的专科医生，只看自己最擅长的病。了解谁负责什么，你就知道该把什么任务交给谁。

六个关键角色

在 30 个 Agent 中，这 6 个是你最常打交道的：

📋

Planner（规划师）

接收需求后拆解成可执行的步骤，标注依赖关系和风险。它是项目的"总导演"，不写代码只做计划。

🔍

Code Reviewer（代码审查员）

逐行检查代码质量、安全漏洞和最佳实践违规。像一位严厉但公正的校对编辑。

🧪

E2E Runner（测试执行员）

自动生成并运行端到端测试，确保你的功能在真实场景中不会出问题。

🔧

Build Resolver（构建修复员）

当构建失败时自动介入，分析错误日志并修复问题。它是编译错误的"急救医生"。

🛡️

Security Reviewer（安全审计员）

专门扫描 漏洞、敏感信息泄露和 供应链风险。

📚

Doc Updater（文档维护员）

当代码变更时自动更新相关文档，保证文档和代码永远同步。

一个 Agent 是怎么定义的

每个 Agent 就是一个 Markdown 文件，包含 YAML Frontmatter 头部：

CODE 

---
name: planner
description: Expert planning specialist
tools: ["Read", "Grep", "Glob"]
model: opus
---

## Your Role
- Analyze requirements
- Create implementation plans
中文解读

--- 开始 YAML 元数据区域

名字叫 planner（规划师）

一段描述，说明这个 Agent 擅长什么

允许使用的工具：读文件、搜索文件、按模式匹配文件 -- 注意它不能写文件！规划师只看不改

使用 Opus 模型 -- 规划需要强推理能力，所以用最贵的模型

--- 结束 YAML 区域

下面是自由格式的指令，告诉 Agent 具体怎么做

分析需求

创建实施计划

💡
关键设计：最小权限原则

注意 Planner 只有 Read/Grep/Glob 权限，没有写入权限。这是 最小权限原则的体现 -- 规划师只负责分析，不应该直接改代码。
