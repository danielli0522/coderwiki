ComfyUI 内部架构
01

从一张图说起：ComfyUI 的工作全貌

当你在画布上点下 Queue Prompt，背后究竟发生了什么？

你用过 ComfyUI，但你知道它是什么吗？

ComfyUI 是当下最流行的 AI 图像生成引擎 之一。你拖拽节点、连上连线、输入一段文字描述，然后点下 Queue Prompt——一张图就出来了。

但这一秒到下一秒之间，发生了非常复杂而优雅的事情。这个课程就是带你走进那段时间里发生的一切。

💡
为什么要学这些？

因为理解了 ComfyUI 的内部机制，你就能：设计更高效的工作流、快速定位报错节点、给 AI 助手更精准的指令、甚至自己写 自定义节点。

ComfyUI 能做什么？

ComfyUI 不只是一个画图工具。它是一个通用的 AI 内容创作流水线：

🎨

图像生成

支持 Flux、SDXL、SD3.5 等主流模型，文生图、图生图、ControlNet 精准控制。

🎥

视频生成

通过 AnimateDiff、SVD 等扩展，把静态图变成视频片段。

🎵

音频 & 3D

通过 API 节点和社区扩展，还能做音频生成和 3D 模型生成。

🛠

无限扩展

数千个社区自定义节点，涵盖后处理、批处理、图像融合、API 调用等。

源码长什么样？

ComfyUI 的核心代码并不庞大。下面是关键文件的地图——这门课的所有内容都围绕这几个文件展开：

ComfyUI/ 项目根目录
main.py 启动入口：初始化服务器、加载节点、启动工作线程
server.py HTTP + WebSocket 服务端：接收 prompt、管理队列、推送进度
execution.py 执行引擎：验证、拓扑排序、逐节点执行、缓存管理
nodes.py 内置节点定义：CLIPTextEncode、KSampler、VAEDecode 等
comfy/ 核心算法层：模型加载、采样器、ControlNet 等
comfy_execution/ 执行基础设施：图结构、缓存策略、验证逻辑
folder_paths.py 模型文件路径管理：checkpoints、loras 等都注册在这里
custom_nodes/ 社区自定义节点：数千个扩展功能的插件

节点是怎么注册进系统的？

每个节点都是一个 Python class。ComfyUI 用一个全局字典 NODE_CLASS_MAPPINGS 来管理它们：

CODE 

class CLIPTextEncode(ComfyNodeABC):
    @classmethod
    def INPUT_TYPES(s):
        return {"required": {
            "text": (IO.STRING, {"multiline": True}),
            "clip": (IO.CLIP, {})
        }}
    RETURN_TYPES = (IO.CONDITIONING,)
    FUNCTION = "encode"
    CATEGORY = "conditioning"

    def encode(self, clip, text):
        tokens = clip.tokenize(text)
        return (clip.encode_from_tokens_scheduled(tokens),)
大白话

定义一个叫 CLIPTextEncode 的节点类...

声明它需要什么输入（这是一个类方法）...

返回输入规格表...

必须输入：一段文字（支持多行编辑）...

还必须输入：一个 CLIP 模型实例...

这个节点输出一个 conditioning（编码后的语义向量）...

核心处理函数名叫 encode...

在节点菜单里归类到 conditioning 分组...

执行编码：先把文字拆成 token（词元）...

然后用 CLIP 模型把 token 编码成语义向量并返回...

💡
通用模式

这个 INPUT_TYPES + RETURN_TYPES + FUNCTION 的三件套是 ComfyUI 节点的标准模板。几乎所有节点（包括你写的自定义节点）都遵循这个约定。学会这个模式，你就能读懂任何一个节点。

概念速查

NODE_CLASS_MAPPINGS 全局字典，把节点类名映射到实际的 Python class 对象——是节点的'花名册'
INPUT_TYPES 类方法，声明节点需要什么输入（类型、默认值、约束）
RETURN_TYPES 元组，声明节点输出什么类型的数据——决定了连线时的类型匹配
FUNCTION 字符串，指定真正执行逻辑的方法名——ComfyUI 会调用这个名字的方法
CATEGORY 节点在菜单中的分类路径，如 "conditioning" 或 "image/upscaling"

03

数据在节点间流淌：执行引擎揭秘

从验证到执行，一条 prompt 走过的完整旅途

为什么需要执行引擎？

你画的节点图可能有几十个节点，连成复杂的网络。但计算机不能'同时'执行所有节点——就像地铁线路图上，列车必须按顺序从起点开到终点。

执行引擎的核心工作就是：确定顺序，按序执行，传递数据。

把执行引擎想象成一个快递分拣中心。每个包裹（数据）都有明确的目的地（下游节点），分拣员（引擎）要确保：

1
验证

包裹上的地址格式对不对？收件人存在吗？（类型检查、循环依赖检测）

2
排序

哪些包裹要先送？不能让收件人等不到上游的包裹。（拓扑排序）

3
执行

按排好的顺序逐个节点执行，把输出传给下游节点，直到所有输出节点完成

4
缓存

上次已经算过的结果如果输入没变，直接复用，不用重新算（智能缓存）

第一步：验证 Prompt

validate_prompt 函数是第一道关卡。它做什么？

CODE 

async def validate_prompt(prompt_id, prompt, partial_execution_list):
    outputs = set()
    for x in prompt:
        class_type = prompt[x]["class_type"]
        class_ = nodes.NODE_CLASS_MAPPINGS.get(class_type)
        if class_ is None:
            return (False, error, [], {})
        if hasattr(class_, "OUTPUT_NODE"):
            outputs.add(x)
    if len(outputs) == 0:
        return (False, error, [], {})
大白话

定义异步验证函数，接收 prompt ID 和节点字典...

准备一个集合来记录哪些节点是'输出节点'...

遍历 prompt 里的每一个节点...

取出这个节点的类型名称...

在全局注册表里查找这个类型——如果找不到...

说明自定义节点没安装，直接返回错误...

如果这个节点标记了 OUTPUT_NODE = True...

就把它记到输出集合里（SaveImage 就是输出节点）...

如果整个图连一个输出节点都没有...

返回错误：这个图没有终点，执行了也没意义。

⚠
常见报错

如果你看到 "Node 'XXX' not found. The custom node may not be installed"——这就是 validate_prompt 发现一个节点的 class_type 在 NODE_CLASS_MAPPINGS 里找不到。解决方案：安装对应的自定义节点。

智能缓存：不重复造轮子

如果一张图你生成两次，但只改了最后的 SaveImage 参数，前面的模型加载、文本编码都不需要重新算。ComfyUI 提供了多种缓存策略：

🔄

Classic 缓存

用完即丢——最保守的策略，每次执行都重新计算。适合调试场景。

📦

LRU 缓存

最近用过的留着，很久没用的丢掉。通过 --cache-lru N 设置大小。

💸

RAM Pressure 缓存

默认策略。根据可用内存动态决定保留多少缓存，内存紧张时自动清理。

CODE 

class CacheType(Enum):
    CLASSIC = 0
    LRU = 1
    NONE = 2
    RAM_PRESSURE = 3
大白话

用枚举定义四种缓存策略...

Classic：经典模式，用完就清...

LRU：保留最近使用的，丢弃最久没用的...

NONE：完全不缓存，每次都从头算...

RAM_PRESSURE：根据内存压力动态调整，默认选择。

04

模型管家：显存、缓存与智能调度

GPU 显存是稀缺资源，ComfyUI 如何聪明地管理它？

显存：AI 绘图最稀缺的资源

生成一张 AI 图片需要加载好几个模型——一个主模型（几 GB）、一个文本编码器（几百 MB）、一个 VAE 解码器——全都得塞进 GPU 显存 里。

如果你的显卡只有 8GB 显存，而模型需要 10GB，就会报 OOM（Out of Memory） 错误。

ComfyUI 的 model_management 模块就像一个精明的仓库管理员，决定哪些模型该上架、哪些该先收起来。

当你拖入一个 CheckpointLoader 节点并执行时：

MM
