Archon 中的巧妙设计

好的软件就像精心设计的机器——每个零件都有它的道理。Archon 中有几个特别巧妙的设计模式值得学习。理解这些模式不仅能帮你更好地使用 Archon，还能让你在指挥 AI 写代码时提出更专业的需求。

模式一：迭代优化循环

Archon 不满足于一次就写出完美代码。它采用「写代码 → 审查 → 修改 → 再审查」的循环模式，就像写作文时的多轮修改——初稿往往不够好，修改几遍后质量才会上去。

CODE 

def should_continue(state):
    if state["iterations"] >= MAX_ITER:
        return "finish"
    if "APPROVED" in state["review"]:
        return "finish"
    return "continue"
PLAIN ENGLISH

定义一个函数来决定是否继续优化。

如果迭代次数达到上限...

就结束循环，返回最终结果。

如果审查者认为代码通过了...

也结束循环。

否则，继续返回编码者优化代码。

💡
通用智慧：迭代优于完美

这个模式在软件工程中极其常见：迭代改进比一次做到完美更现实。不管是写代码、写文章还是做设计，「先出初版，再逐步优化」永远是更高效的策略。

模式二：防御性编程

Archon 的代码中处处体现了「防御性编程」的思想——假设任何可能出错的地方都会出错，提前准备好应对措施。就像一个谨慎的司机，不仅系好安全带，还买了保险、备了备胎。

CODE 

try:
    response = llm.invoke(prompt)
except RateLimitError:
    time.sleep(60)
    response = llm.invoke(prompt)
except Exception as e:
    return {"error": str(e)}

如果遇到其他类型的错误...

返回错误信息，而不是让程序崩溃。

当 Agent 出错了怎么办？

不管代码写得多好，问题总会出现。学会排查问题是使用 AI 工具最重要的能力之一。以下是常见的出错场景和排查方法：

🔴

Agent 无限循环

症状：Agent 一直在重复同样的操作，不停下来。排查：检查 should_continue 函数的退出条件是否正确。可能 MAX_ITER 设得太高，或者审查者永远不会返回 APPROVED。

🔵

生成代码有语法错误

症状：生成的代码无法运行。排查：检查审查者节点是否正常工作，或者 prompt 是否足够清晰。有时候给 LLM 提供一个代码示例就能大幅提高生成质量。

🟢

API 调用超时

症状：Agent 运行到某一步就卡住了。排查：查看网络连接和 API 密钥。增加重试逻辑（如上面的代码示例）可以解决大部分问题。
