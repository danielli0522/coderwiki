插件架构：一段配置看清全貌

ECC 的核心是一份 配置文件 plugin.json。它告诉 Claude Code："加载哪些 Hook、哪些 Agent、哪些 Command。"

plugin.json { "name": "everything-claude-code", "version": "2.0.0-rc.1", "description": "The ultimate Claude Code framework", "hooks": ["hooks/hooks.json"], "agents": ["agents/*.md"], "commands": ["commands/*.md"], "skills": ["skills/*.md"] }
中文解读
name — 插件的名字叫 "everything-claude-code"，即"万物 Claude Code"
version — 当前版本号 2.0.0-rc.1，rc 表示"发布候选版"
description — 一句话描述："终极 Claude Code 框架"
hooks — 指向 hooks.json，里面注册了各种自动化钩子
agents — 用通配符 *.md 加载 agents 目录下所有 60 个 Agent 定义
commands — 加载 commands 目录下所有 75 个命令定义
skills — 加载 skills 目录下所有 232 个技能定义

注意 *.md 这个通配符写法 — 它的意思是"加载这个目录下所有的 Markdown 文件"。每个 Agent、Command、Skill 都是一个独立的 .md 文件，这种设计让你可以灵活增减功能。

🔍
设计亮点：配置与代码分离

plugin.json 里的 hooks 字段指向的是 hooks/hooks.json（另一个配置文件），而不是直接写 Hook 代码。 这种"配置文件指向配置文件"的设计，让每个部分都能独立修改，互不影响。

02

五大角色

一部电影的幕后，由五种角色共同撑起。Claude Code 也是如此。

电影剧组的五种角色

ECC 的世界里，有五类「演员」在幕后协作。它们各有分工，共同确保每一次 prompt 都能得到精准响应。

D
Agent — 导演

做决策、分解任务、委派给其他角色。它是整个剧组的指挥中心。

S
Skill — 编剧

预写好的「剧本」，让 AI 面对常见场景时不用每次从零开始思考。

C
Command — 道具师

用户随时可以调用的斜杠命令，像道具一样即拿即用。

H
Hook — 舞台监督

在幕后默默运行，特定事件发生时自动介入。你不需要手动调用它。

I
Instinct — 演员记忆

从过往会话中学到的经验教训，帮助 AI 避免重复犯错、越用越聪明。

Aha!
为什么是五个而不是三个？

很多人以为 AI 只需要「理解 + 回答」。但 ECC 把能力拆成五个角色，就像电影不能只有导演和演员——编剧写剧本、道具师准备工具、舞台监督管流程、演员积累经验。每个角色专注一件事，组合起来才能应对复杂场景。这就是配置即能力的精髓。

深入 Agent：导演的剧本

一个 Agent 的配置文件 长什么样？下面是一段真实的 agents/planner.md 代码。

markdown --- name: planner description: > Strategic planning agent that breaks down complex tasks into actionable steps. Coordinates with other agents and delegates subtasks. directives: - Always think before acting - Break complex problems into steps - Delegate to specialized agents ---
中文解读
name: planner — 这个 Agent 叫「规划师」
description — 它是战略规划型代理，负责把复杂任务拆解为可执行的步骤，并与其他 Agent 协调
directives — 行为准则（导演的执导原则）：
  1. 行动前先思考
  2. 把复杂问题分解成步骤
  3. 把子任务委派给专业 Agent

Markdown 文件开头的 --- 区域叫 frontmatter， 用 YAML 语法定义了 Agent 的「身份」和「行为准则」。

Aha!
directives 就是导演的「执导原则」

注意那三条 directives——它们不是代码逻辑，而是用自然语言写的「行为约束」。AI 会把这些原则内化为决策依据。你写什么，它就倾向什么。这就是 directive 的力量。

03

会话生命周期

想象你入住一家酒店 —— 办理入住、享受住宿、退房结账。ECC 的会话管理也是这个流程，只不过"忠诚积分"让每次入住都比上一次更丝滑。

酒店入住 —— session-start 的五步仪式

你打开 Claude Code，ECC 的 session-start hook 就像前台接待员一样自动开始工作。它会帮你取出上次的"行李"，准备好所有服务。

整个过程不到一秒，但完成了五件关键的事：

1
Claude Code 启动，session-start.js 触发

Hook 机制监听到 session 启动事件，自动调用入口脚本。你什么都不用做。

2
加载上一次会话的摘要

从 持久化 存储中读取上次 context，让你从上次中断的地方继续。

3
恢复积累的"本能"（instincts）

加载你过去所有会话中提炼出的 instinct —— 你的编码偏好、常用模式、已知项目风格。

4
注册所有 hooks 和 agents

把 agent 和 checkpoint 等所有扩展能力挂载到当前会话的 lifecycle 中。

5
你可以开始编码了！

ECC 已经把"老客户的档案"全部准备就绪。你的 token 配置、历史偏好、安全策略都已生效。

session-start.js —— 读代码，看前台做了什么

下面是 scripts/hooks/session-start.js 的核心逻辑。左边是代码，右边是它每一行在"做什么事"：

CODE · session-start.js 
async function onSessionStart() {
  const context = await loadSessionContext();
  const instincts = await loadInstincts();
  const skills = await discoverSkills();

  if (context.lastSession) {
    console.log(`Resuming from: ${context.lastSession.id}`);
  }

  await registerHooks(hookConfigs);
  await loadAgents(agentPaths);

  return { context, instincts, skills };
}
中文翻译

定义异步函数 onSessionStart —— 会话启动时的总入口。

从磁盘加载上一次会话的 context（对话历史、项目信息）。

加载所有积累的 instinct（你的编码偏好和学到的模式）。

扫描 skillpack 目录，发现所有可用的 skill。

如果存在上一次会话记录的话...

打印日志：正在从某个会话 ID 恢复。

（条件结束）

把所有 hook 配置（安全、学习、检查点等）注册到生命周期中。

加载所有 agent 模块（安全 agent、学习 agent 等）。

返回打包好的三件套：context、instincts、skills，供后续使用。

📝
为什么要分三步加载？

context 是"你上次在做什么"，instincts 是"你一直喜欢怎么做"，skills 是"现在有什么工具可以用"。三者更新频率不同：context 每次会话变，instincts 偶尔积累更新，skills 只在你安装新技能包时才变。分开存储，关注点分离，读写更高效。

持续学习引擎

想象你在健身房办了一张会员卡 —— 每次锻炼（会话）都在积累肌肉记忆（本能）。去的次数越多，力量越强。置信度分数就像你的个人记录 tracker，精确衡量每一次进步。

为什么持续学习很重要？

普通的 Claude Code 每次启动都是一张白纸 —— 没有任何记忆，不认识你，不了解你的项目。

ECC 的 instinct 系统让 Claude 从你的每一次交互中学习，越用越懂你。

🎨
代码风格偏好

学习你的格式化选择 —— 缩进用 tab 还是空格、分号加不加、单引号还是双引号。ECC 的 pattern recognition 让它自动适配你的风格。

🏗️
项目结构认知

记住你的架构决策 —— 组件放哪个目录、状态管理用什么方案、API 路由怎么组织。就像健身教练了解你的身体结构一样。

🔍
错误模式识别

更快发现重复出现的问题。如果你总是忘记处理某个 edge case，ECC 会主动提醒你。

🛠️
工具使用习惯

优化工具选择 —— 你更喜欢用 grep 还是 find？习惯先跑测试还是先写代码？ECC 会学着像你一样思考和行动。

💡
Aha! — 普通 AI 像金鱼

普通 AI 像金鱼，每次对话都是全新的。ECC 让 AI 像你的私人教练，记得你上次练了什么、哪些动作做得好、哪里还需要加强。

关键区别在于 feedback loop：每次你确认或纠正 ECC 的行为，它都在学习。这就是 machine learning 思想在开发工具中的落地。

Instinct 系统 —— 读代码，看"肌肉记忆"怎么存

每个 instinct 都是一个结构化对象。左边是代码，右边是它在"记录什么"：

javascript 
const instinct = {
  pattern: "prefer-typescript-strict-mode",
  confidence: 0.92,
  source: "observed-across-8-sessions",
  lastUpdated: "2024-01-15",
  appliesTo: ["new-files", "refactoring"]
};

if (instinct.confidence >= confidenceThreshold) {
  applyInstinct(instinct);
}
中文解读
pattern — 模式名称：这个用户偏好 TypeScript 严格模式
confidence — confidence score = 0.92，即 92% 的把握。经过 8 次会话观察得出。
source — 数据来源：在 8 个会话中反复观察到这个偏好
lastUpdated — 最后更新时间，确保使用最新数据
appliesTo — 适用范围：创建新文件和重构代码时才生效
 
如果置信度 >= threshold 阈值...
就自动应用这个 instinct —— 像肌肉记忆一样，不需要你再次确认。
🔍
为什么用 0.92 而不是 100%？

置信度永远不会到 1.0，因为人的偏好可能随时改变。0.92 代表"我非常确定，但留了 8% 的余地"。如果连续几次你开始用 JavaScript 而不是 TypeScript，置信度会自动下降。这就是 heuristics 的智慧 —— 不追求绝对正确，追求持续适应。

05

安全防线

机场安检的 X 光机扫描行李，AgentShield 扫描代码。每一次 AI 操作都要过安检，才能抵达目的地。

AI 编程的安全威胁 —— 为什么机场安检不是摆设？

AI 编程工具能执行代码、修改文件、访问 API — 能力越大，风险越大。

最大的威胁叫 Prompt Injection ：恶意指令藏在代码注释、文件名或数据里，AI 一不小心就会执行。

ECC 的 AgentShield 就像机场安检系统 — X 光扫描仪（扫描威胁）、护照检查（验证身份）、禁飞名单（拦截已知危险模式）、安全须知（提升用户意识）。

1
恶意代码注入

攻击者在 prompt 中藏入 eval()、exec() 等指令，诱导 AI 执行危险操作。就像行李箱里藏了违禁品。

2
文件系统越权

AI 读写了不该碰的文件 — .env 密钥、系统配置、用户隐私数据。就像有人误入了机场的禁区。

3
数据泄露

AI 把 DATABASE_URL、API 密钥等敏感信息输出到日志或代码中。就像安检时你的护照信息被广播了出来。

4
供应链攻击

让 AI 安装了一个被篡改的 依赖包 ，攻击者通过它渗透你的项目。就像通过了安检的包裹里其实藏着违禁品。

AgentShield 配置 —— 读代码，看安检机器怎么运转

AgentShield 的核心是 PreToolUse Hook — 它在 AI 每次使用工具前拦截，就像 X 光机在行李上飞机前扫描。

hooks/hooks.json { "PreToolUse": [{ "matcher": "Write|Edit", "hooks": ["scripts/hooks/security-scan.js"], "options": { "scanPatterns": [ "eval(", "exec(", "child_process", "fs.unlink", "rm -rf" ], "blockOnThreat": true, "logLevel": "warn" } }] }
中文解读
PreToolUse — 在 AI 使用工具"之前"触发，这是安检的第一道闸门
matcher: "Write|Edit" — 只监听"写文件"和"编辑文件"这两种操作，因为它们最容易注入恶意代码
hooks — 指向安全扫描脚本，它就是那台 X 光机
scanPatterns — 禁飞名单！列出所有危险模式：eval()、exec()、child_process
blockOnThreat: true — 发现威胁立刻拦截，不让 AI 执行这个操作（相当于"禁止登机"）
logLevel: "warn" — 把安全事件记录到日志中，方便事后审查（相当于安检的监控录像）
🔍
设计亮点：关注点分离

配置文件（hooks.json）只定义"什么时候触发"和"用什么规则"，具体扫描逻辑在 security-scan.js 中。 这就像安检规定（"哪些东西违禁"）和 X 光机器（"怎么检测"）是分开的一样 — 规则可以随时更新，不用换机器。

Aha! — AI 时代最大的安全威胁

在 AI 编程时代，最大的安全威胁不是黑客，而是你让 AI 处理的代码里藏着的恶意指令。 AgentShield 就像代码世界的杀毒软件，但它扫描的不是病毒，而是 意图。
