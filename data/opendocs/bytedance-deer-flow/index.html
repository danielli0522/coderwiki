DeerFlow 是什么

一个字节跳动开源的超级 Agent 编排框架——把子智能体、记忆、沙箱和技能组合在一起，帮你完成几乎任何任务

想象你在用 DeerFlow

你在浏览器里打开 DeerFlow 的聊天界面，输入一句话：

💬
用户输入

"帮我研究一下 2026 年 AI Agent 框架的市场格局，生成一份带图表的报告"

然后你坐下来喝杯咖啡。几分钟后，你收到了一份完整的分析报告——包含数据搜索、竞品对比、趋势图表。这不是一个 LLM 在单打独斗，而是一整支"AI 团队"在并行工作。

它到底做了什么？

DeerFlow 2.0（Deep Exploration and Efficient Research Flow）是字节跳动开源的 超级 Agent 编排框架。它不是一个聊天机器人，而是一个"AI 团队的项目经理"。

1
接收你的指令

通过网页聊天、飞书、钉钉、Telegram 等渠道输入任务

2
拆解成子任务

Lead Agent 把复杂任务分解成多个可以并行执行的子任务

3
派发子 Agent 执行

每个子 Agent 在独立的沙箱环境里搜索、编码、分析

4
汇总结果返回给你

Lead Agent 把所有子 Agent 的结果整合成一份连贯的回答

💡
关键洞察

DeerFlow 2.0 是一次彻底重写，和 v1 没有共用代码。v1 是一个 Deep Research（深度搜索）框架，v2 进化成了一个通用的"超级 Agent 驾驶舱"——可以接入各种工具、记忆、技能，完成几乎任何任务。

认识核心角色

Gateway、Lead Agent、Sub-Agent、Middleware——每个角色各司其职，像一支配合默契的乐队

乐队成员名单

把 DeerFlow 想象成一支交响乐队。指挥是 Lead Agent，乐手是子 Agent，乐谱是 Skills，舞台管理员是 Gateway。

🚪
Gateway（API 网关）

舞台管理员——检查门票（身份验证）、安排座位（路由分发）、维持秩序（ CSRF 防护）

🦌
Lead Agent（主智能体）

乐队指挥——理解用户意图、决定调用哪些工具、编排子 Agent、汇总最终答案

⚡
Sub-Agent（子智能体）

乐手——被指挥派出去完成具体任务（搜索、编码、分析），在独立沙箱中工作

🔧
Middleware（中间件链）

音效处理器——在消息到达 Agent 之前和之后做各种加工（摘要、记忆、循环检测等）

📦
Sandbox（沙箱）

排练室——子 Agent 在这里安全地执行代码和命令，不会影响外部系统

Agent 是如何被创建的

每次对话开始时，make_lead_agent() 工厂函数会动态组装一个 Agent——根据配置选择模型、加载工具、串起 中间件链：

CODE 

def _make_lead_agent(config, *, app_config):
    model_name = _resolve_model_name(
        cfg.get("model_name"))
    tools = get_available_tools(
        model_name=model_name)
    return create_agent(
        model=create_chat_model(name=model_name),
        tools=tools,
        middleware=_build_middlewares(config),
        system_prompt=apply_prompt_template(),
    )
中文解读

工厂函数入口，接收运行时配置和应用配置。

根据用户请求或配置，决定用哪个大模型。

加载所有可用的工具（文件操作、搜索、代码执行等）。

用 LangChain 的 create_agent 创建最终的 Agent 实例。

注入选定的大模型。

注入工具集。

构建并注入中间件链（14 层处理！）。

应用系统提示词模板——告诉 Agent 它是谁、能做什么。

14 层中间件：请求的质检流水线

每个用户消息在被 Lead Agent 处理之前，都要经过一条精心设计的中间件链。每层中间件就像流水线上的一个质检站：

中间件链（按执行顺序）
ThreadDataMiddleware 绑定当前会话的线程信息
SandboxMiddleware 初始化沙箱环境
DanglingToolCallMiddleware 修复缺失的工具调用响应
SummarizationMiddleware 对话太长时自动摘要压缩
TodoMiddleware 管理任务清单（规划模式）
TitleMiddleware 自动给对话生成标题
MemoryMiddleware 把对话内容存入长期记忆
LoopDetectionMiddleware 检测 Agent 是否陷入死循环
ClarificationMiddleware 遇到不明确的请求，主动向用户提问
⚠️
中间件顺序很重要

代码注释里明确写道：ClarificationMiddleware 必须在最后，因为它需要在所有其他中间件处理完之后才能判断是否需要向用户提问。如果你在开发自己的中间件，要注意插入位置。

async def lifespan(app: FastAPI):
    startup_config = get_app_config()
    apply_logging_level(
        startup_config.log_level)
    async with langgraph_runtime(
        app, startup_config):
        await _ensure_admin_user(app)
        channel_service = \
            await start_channel_service()
        yield

设计精妙之处

系统提示词（System Prompt）是静态的，而日期和记忆通过 DynamicContextMiddleware 作为 <system-reminder> 注入到第一条 HumanMessage 中。这样做的目的是最大化前缀缓存（prefix caching）的命中率——所有用户共享同一段系统提示词，只有动态部分不同。

子 Agent 调度：并行执行的秘密

当 Lead Agent 决定拆分任务时，它通过 task 工具调用 SubagentExecutor。这段代码展示了子 Agent 如何被创建和执行：

CODE 

MAX_CONCURRENT_SUBAGENTS = 3

class SubagentExecutor:
    def execute_async(self, task):
        task_id = uuid.uuid4()[:8]
        result = SubagentResult(
            task_id=task_id,
            status=SubagentStatus.PENDING)
        _scheduler_pool.submit(
            run_task)
中文解读

硬性并发上限：最多 3 个子 Agent 同时运行。超出部分会被静默丢弃！

子 Agent 执行器——每个子任务创建一个独立的执行实例。

异步执行入口：不阻塞主 Agent，在后台线程池中运行。

为每个子任务生成唯一 ID，用于追踪状态。

创建结果容器——记录任务状态、开始/结束时间、token 消耗。

提交到线程池（最多 3 个调度线程），开始后台执行。

智能扩展机制

Skill 技能系统、Memory 长期记忆、Sandbox 沙箱执行——DeerFlow 的三大扩展支柱

三根支柱撑起整个系统

如果把 DeerFlow 比作一棵树，Lead Agent 是树干，那么 Skill、Memory 和 Sandbox 就是三根主要的树枝——分别负责"能力"、"经验"和"安全"。

🎯

Skills 技能系统

像给 AI 装"技能包"——深度研究、PPT 生成、代码文档、数据分析。每个技能包含最佳实践和工作流模板。

🧠

Memory 长期记忆

像日记本——Agent 记住你的偏好、项目上下文、历史对话要点。下次对话不需要重复说明。

🔒

Sandbox 沙箱执行

像隔离实验室——子 Agent 在 Docker 容器里执行代码、安装包、操作文件，不会影响宿主系统。

Skills：Agent 的技能背包

每个 Skill 就是一个 SKILL.md 文件，包含了这个任务的最佳实践、工作流程和参考资料。DeerFlow 内置了 20+ 个技能：

deep-research 深度研究——多轮搜索 + 信息综合 + 报告生成
ppt-generation PPT 生成——从文本大纲到完整的演示文稿
data-analysis 数据分析——读取 CSV/Excel，生成图表和洞察
code-documentation 代码文档——自动为代码库生成文档
skill-creator 技能创建器——让 Agent 自己创建新的技能包
podcast-generation 播客生成——把研究内容转成播客脚本

技能加载遵循"渐进式加载"模式：Agent 先看到技能列表，匹配到相关技能后才加载完整内容——这就像你不会一次打开所有 App，而是按需打开。

CODE 

def get_skills_prompt_section(
    available_skills):
    skills = get_enabled_skills()
    skill_items = "\n".join(
        f"<skill><name>{s.name}</name>"
        f"<description>{s.description}"
        f"</description></skill>"
        for s in skills)
中文解读

生成技能部分的系统提示词。

先从磁盘加载所有已启用的技能元数据。

把每个技能格式化成 XML 标签，告诉 Agent 有哪些技能可用。

Agent 看到这个列表后，只在需要时才用 read_file 读取完整技能内容。

这避免了把所有技能内容都塞进上下文——节省 token、提高响应速度。

DeerFlow 的记忆不是简单的"把历史聊天存下来"。它有一个结构化的记忆模型，分为多个维度——就像一个人的记忆分为工作记忆、个人经历、长期背景知识：

Sandbox：安全的执行环境

子 Agent 需要执行代码、安装软件包、读写文件——这些操作不能直接在你的服务器上跑。DeerFlow 使用 Docker 容器 作为沙箱：

CODE 

class AioSandbox(Sandbox):
    _MAX_DOWNLOAD_SIZE = 100 * 1024 * 1024
    _DEFAULT_NO_CHANGE_TIMEOUT = 600

    def __init__(self, id, base_url, home_dir):
        self._client = AioSandboxClient(
            base_url=base_url,
            timeout=600)
        self._lock = threading.Lock()
中文解读

AioSandbox 通过 HTTP API 连接到一个 Docker 容器——就像远程操控一台独立的小电脑。

下载文件大小上限 100MB，防止子 Agent 搞出巨型文件占满磁盘。

命令无输出超时 600 秒（10 分钟），防止命令卡死。

初始化时连接到指定的沙箱容器 URL。

API 超时也是 600 秒。

线程锁确保同一时间只有一个命令在执行——因为容器的 shell 是单会话的。

💡
安全边界设计

沙箱有三个挂载目录：/mnt/user-data/uploads（用户上传）、/mnt/user-data/workspace（工作目录）、/mnt/user-data/outputs（最终产出）。子 Agent 只能在这个范围内活动。

适用场景分析

✅ 适合

  • ✓ 需要多步骤研究的任务（市场分析、竞品调研）
  • ✓ 需要代码执行的 AI 助手（数据分析、自动化脚本）
  • ✓ 企业内部 AI 助手（对接飞书/钉钉）
  • ✓ 需要自定义 Agent 人格和技能的场景
  • ✓ 技术团队想快速搭建可扩展的 Agent 平台

❌ 不适合

  • ✗ 只需要简单对话问答（杀鸡用牛刀）
  • ✗ 无 Docker 环境的轻量部署场景
  • ✗ 需要实时低延迟响应（子 Agent 有启动开销）
  • ✗ 纯前端应用（DeerFlow 是后端重架构）
  • ✗ 对 LLM 成本极度敏感（多 Agent 调用消耗更多 token）

源码中的硬限制

这些是从源码中提取的实际常量——了解它们能帮你判断 DeerFlow 是否满足你的规模需求：

MAX_CONCURRENT_SUBAGENTS = 3 每个 Lead Agent 响应最多并行 3 个子 Agent，超出部分被静默丢弃
_MAX_DOWNLOAD_SIZE = 100MB 沙箱中单个文件下载上限 100MB
_DEFAULT_NO_CHANGE_TIMEOUT = 600s 沙箱命令无输出超过 10 分钟自动终止
_scheduler_pool max_workers = 3 后台调度线程池大小固定为 3
_ENABLED_SKILLS_REFRESH_WAIT = 5s 技能缓存刷新等待超时 5 秒

成本考量与结论

1
LLM Token 成本

每次用户请求可能触发 1 次 Lead Agent 调用 + 最多 3 次子 Agent 调用，每次都消耗 token。建议用 DeepSeek 或 GLM 等低成本模型。

2
基础设施成本

需要 Docker 环境运行沙箱，最低配置 4GB RAM。推荐 8GB+ 以支持并行子 Agent。

3
运维复杂度
