主要角色：agno 里谁负责什么

把系统想成一个协作团队，避免让 AI 把逻辑塞错位置。

四类角色

入口

CLI、Web UI、API 或 SDK，负责把人的意图变成结构化请求。

编排

把任务拆开，决定调用模型、工具、索引还是插件。

能力

模型推理、网页抓取、向量搜索、文件读写、浏览器调试等实际能力。

状态

配置、缓存、会话、向量库、任务历史，决定系统能不能稳定复盘。

代码翻译：把输入变成可执行步骤

CODE
request = normalize(raw_input)
context = load_context(request.session_id)
plan = planner.create(request, context)
response = executor.run(plan)
中文解释

normalize 负责把混乱输入变成系统认识的格式。

load_context 取出会话、配置或索引等上下文。

planner 生成可执行计划，executor 才真正执行。

如果结果不对，就沿这四行逐层打印中间值。

实践应用：怎样把 agno 接进自己的项目

把开源库变成可控能力，而不是一次性 demo。

接入前的三张清单

输入契约

你的业务会传什么字段，哪些字段必须校验，失败时返回什么。

运行边界

是否需要 GPU、浏览器、网络、密钥、数据库或持久化目录。

观测证据

日志、trace、任务状态、成本统计，决定上线后能不能查问题。

代码翻译：最小适配层

CODE
async function callagno(input) {
  const payload = validateInput(input);
  const result = await client.run(payload);
  return mapToProductResult(result);
}
中文解释

外面包一层适配函数，不让业务代码直接依赖第三方细节。

validateInput 先把不合法输入挡住。

client.run 调用 agno 的核心能力。

mapToProductResult 把返回值变成产品自己的格式。

上线风险

!
别只跑通 happy path

AI 项目最容易忽略三类失败：模型超时、外部工具返回脏数据、状态目录权限错误。让 AI 写接入代码时，要明确要求这些失败都可见、可重试、可回滚。

代码翻译：选型要落到接口

CODE
interface AdoptionDecision {
  repo: "agno-agi/agno";
  fit: "adopt" | "pilot" | "avoid";
  risks: string[];
  verification: string[];
}
中文解释

repo 记录本次评估的开源项目，避免口头判断丢失上下文。

fit 不是简单喜欢或不喜欢，而是采用、试点、暂缓三种决策。

risks 写明部署、成本、协议、性能和维护风险。

verification 写出真正要跑的验证命令和验收页面。

适合 / 不适合

适合
  • 团队愿意阅读源码并做小步集成。
  • 需要掌控模型、工具、数据或运行环境。
  • 希望把 AI 能力变成可复用基础设施。
不适合
  • 只想要一次性演示，不准备维护。
  • 没有日志、权限、密钥和成本治理意识。
  • 业务需求和项目定位完全不匹配。
