01

给代码装上"神经系统"

GitNexus 是什么？为什么 40k+ 开发者给它点了 Star？

AI 代理的致命盲区

你让 Cursor 修改 UserService.validate()，它改了返回类型。看起来没问题。但它不知道：整个项目里有 47 个函数依赖这个返回类型。上线后，47 个地方全部报错。

这不是 AI 不够聪明，而是它根本看不到代码之间的依赖关系。它只能一个文件一个文件地 grep、read、guess——就像蒙着眼在迷宫里找路。

💡
核心问题

Cursor、Claude Code、Codex 等 AI 工具能读写代码，但不真正理解代码库的结构。它们不知道谁调了谁、改一个函数会影响多少地方。GitNexus 就是来补这块拼图的。

GitNexus：代码的"神经系统"

GitNexus 把你的整个代码库解析成一张 知识图谱——每个函数、每个类、每条调用链、每个依赖关系，全部预先计算好，存在本地图数据库里。AI 代理一次查询就能拿到完整的上下文。

🕵

预计算关系智能

索引时就把聚类、执行流追踪、影响范围评分全部算好。查询时一次返回完整上下文，LLM 不可能遗漏。

💰

Token 极省

传统方式需要 10+ 次 grep/read 查询才能理解一个函数。GitNexus 一次查询就给完整答案，省掉 70%+ 的工具调用。

🎯

小模型也能用

工具做了重活，LLM 只需要读懂结构化结果。即使较小的模型也能获得完整的架构视野。

🔒

完全本地运行

CLI 模式全部本机运行，不联网；Web 模式全部浏览器内运行，不上传代码。你的代码永远不离开你的机器。

两种使用方式，覆盖所有场景

GitNexus 提供两种互补的使用模式：

CLI + MCP 模式

02

Zero-Server 架构：代码不出浏览器

为什么 GitNexus 能做到"代码永远不离开你的机器"？

两个包，一套逻辑

GitNexus 是一个 monorepo，包含两个核心包和共享类型库：

CLI
gitnexus/ — 核心引擎

CLI 命令行工具 + MCP 服务器 + 索引管道 + 图数据库。用 Tree-sitter 原生绑定解析代码，LadybugDB 原生模式存储图谱。Node.js 运行时。

WEB
gitnexus-web/ — 浏览器界面

React 18 + Vite + Tailwind v4 构建的可视化图谱浏览器。Tree-sitter WASM 解析代码，LadybugDB WASM 存储图谱，WASM 模式运行嵌入模型。一切在浏览器里完成。

SHARED
gitnexus-shared/ — 共享类型

TypeScript 类型和常量，CLI 和 Web 都用。支持 14 种语言的枚举、图谱 schema 定义、搜索接口等。

Zero-Server 理念的三个层次

🔒
隐私优先

CLI 模式全部本机运行，零网络请求。Web 模式全部浏览器内完成，API Key 只存在 localStorage。代码永远不离开你的机器。

🌐

解析层 — Tree-sitter WASM

Web 版把 Tree-sitter 编译成 WASM 在浏览器里运行。14 种语言的语法解析，全部在用户浏览器里完成，不调用任何外部 API。

🗃

存储层 — LadybugDB WASM

图数据库也编译成 WASM，在浏览器内存中运行。每次会话独立，关掉标签页数据自动清除。CLI 版则用原生 LadybugDB，持久化到 .gitnexus/ 目录。

🤖

嵌入层 — transformers.js

用 HuggingFace 的 transformers.js 在浏览器里生成文本嵌入向量（WebGPU 加速或 WASM 降级）。CLI 版用 Node.js 原生版本，支持 GPU 加速。

Bridge 模式：本地最强组合

CLI 和 Web 不是割裂的。通过 Bridge 模式，你可以两全其美：

12 阶段管道：从源码到知识图谱

一行 gitnexus analyze 背后发生了什么？

代码索引就像石油炼化

源文件（原油）进来后，先扫描（探明储量），再解析（蒸馏出符号），然后关联（调和引用关系），聚类（分类存储），追踪（画出调用流程图），最终入库（写入图数据库）。每一道工序都依赖上游的结果，形成 DAG 式的依赖关系。

12 阶段 DAG 

// 阶段依赖关系（简化版）
scan → structure → [markdown, cobol]
  → parse → [routes, tools, orm]
  → crossFile → mro → communities
  → processes
大白话

scan：扫描文件树，记录所有文件路径和大小。

structure：建立文件夹和文件的层级关系。

parse：用 Tree-sitter 解析语法树，提取函数、类、方法等符号。

crossFile：跨文件传播类型信息，解析 import 链。

mro：解析方法解析顺序（继承链上的方法覆盖关系）。

communities：用 Leiden 算法把相关符号聚成功能社区。

processes：从入口点开始追踪执行流（如 HTTP 请求到响应的完整调用链）。

核心：Tree-sitter 解析引擎

parse 阶段是整个管道的心脏。它用 Tree-sitter 把每个源文件解析成 AST，然后用统一的捕获标签提取符号。

1
语言检测与查询

根据文件扩展名检测语言（.ts → TypeScript, .py → Python, .rs → Rust），加载对应的 Tree-sitter 语法。关键：不同语言用不同的 AST 节点名，但产出同一套语义标签（@definition.class, @call.name, @import.source）。

2
分块并行解析

文件按 ~2MB 字节预算分块，通过 Worker 线程池并行解析。每个 Worker：检测语言 → 加载语法 → 运行查询 → 返回统一结果。CLI 最多 16 个 Worker，Web 用 Web Workers + Comlink。

3
符号表与边构建

提取的符号（Function, Class, Method 等）作为图谱节点，import、call、extends 等关系作为边。每条边带有置信度分数（same-file 0.95, import-scoped 0.9, global 0.5）。

6 阶段调用解析：如何知道谁调了谁？

这是 GitNexus 最精巧的部分。当你看到 obj.validate()，GitNexus 需要确定 obj 是什么类型、validate 是哪个类的方法。它通过 6 个阶段的管道来完成：

1-2
提取+分类
3-4
推断+分派
5-6
解析+写入
点击"下一步"追踪调用解析流程

自动发现功能模块和执行流

解析完符号后，GitNexus 还要做两件更高级的事：

🌐

Leiden 社区检测

用 Leiden 算法自动把相关符号聚成功能社区（Authentication、Database、UI 等）。每个社区有一个凝聚度分数，帮助 AI 理解代码库的功能分区。

🔀

执行流追踪

从入口点（HTTP 路由、CLI 命令等）开始追踪完整的调用链。比如一个 LoginFlow 可能追踪 7 步：路由 → 中间件 → 验证 → 查库 → 生成 Token → 返回响应。

🔍

混合搜索

BM25 关键词搜索 + 语义向量搜索，通过 Reciprocal Rank Fusion 融合排序。支持 14 种语言的代码搜索。

Web 界面：交互式图谱浏览器

可视化探索代码结构、AI 对话问答、多语言支持

把代码变成可以"走"的地图

GitNexus Web 用 Sigma.js + Graphology 把知识图谱渲染成交互式网络图。你可以拖拽节点、缩放视图、点击查看详情——就像用 Google Maps 探索城市一样探索代码。

🌍
WebGL 渲染 = 性能保证

Sigma.js 用 WebGL 渲染，GPU 加速。即使图谱有数千个节点和上万条边，拖拽和缩放依然流畅。这是 Canvas 2D 做不到的。

Graph RAG Agent：对着图谱问问题

Web 界面内置一个基于 LangChain ReAct 的 AI 代理。你可以用自然语言提问，它会自动调用图谱工具查找答案：

AI 对话示例

"认证模块是怎么工作的？"

//    认证流程分 7 步：

大白话

你用中文问"认证模块怎么工作的"，不需要知道任何 API。

找到 LoginFlow 后，用 context 工具查看关键函数的上下游。

最后用自然语言总结出完整的认证流程，就像一个熟悉代码的同事在给你讲解。

16 个工具：AI 的代码透视眼镜

GitNexus 通过 MCP 协议暴露 16 个工具，让 AI 代理拥有了前所未有的代码洞察力：

Q
query — 混合搜索

BM25 + 语义向量 + RRF 融合。搜索结果按执行流分组，标注每个符号在流程中的位置。跨社区流程优先展示。

C
context — 360 度符号视图

选中一个函数，立刻看到：谁调它、它调谁、属于哪个社区、参与哪些执行流。就像 LinkedIn 的人际关系全景图。

I
impact — 影响范围分析

修改一个函数前先跑 impact。返回：上游（谁依赖我，会断裂）、下游（我依赖谁，可能连锁崩溃）、风险等级、置信度。

D
detect_changes — 变更检测

提交前跑一下，自动对比 git diff 和图谱，告诉你改了哪些符号、影响了哪些执行流、风险多高。防止"改一行坏一片"。

技术选型与对比：为什么是浏览器端？

GitNexus 的每一个关键选型背后的思考

为什么选择"零服务器"？

大部分代码分析工具都需要服务器：把代码上传到云端解析，返回结果。GitNexus 反其道而行——全部在浏览器或本机完成。为什么？

🔒

企业代码不出墙

企业客户的源代码是核心资产。要求上传到第三方服务器？门都没有。GitNexus 的零服务器架构从根上消除了这个障碍。

⚡

零延迟体验

本地解析 + 本地查询 = 毫秒级响应。不需要等网络传输、不需要排队等服务器资源。WebGL 渲染数千节点丝滑流畅。

💰

零运维成本

不需要维护服务器集群、不需要扩缩容、不需要担心宕机。开源项目也没有服务器账单的压力。

关键技术选型

TS
Tree-sitter 而非 LSP

LSP 虽然语义更丰富，但需要为每种语言启动独立的语言服务器进程，不适合浏览器环境。Tree-sitter 是轻量级的 AST 解析器，能编译成 WASM，14 种语言共享同一套处理逻辑。

DB
LadybugDB 而非 Neo4j

Neo4j 是重量级图数据库，需要 Java 运行时和独立服务器进程。LadybugDB（原 KuzuDB）是嵌入式图数据库，支持向量搜索，能编译成 WASM 在浏览器里跑。原生模式和 WASM 模式共享同一套查询接口。

EM
transformers.js 而非 OpenAI API

用 HuggingFace 的 transformers.js 在本地生成嵌入向量（Snowflake arctic-embed-xs, 384维），不依赖任何外部 API。Web 版用 WebGPU 加速或 WASM 降级，CLI 版支持 Node.js GPU 加速。

MCP
MCP 而非自定义插件协议

MCP 是 AI 工具领域的"USB 接口"。一套 MCP 服务器，Cursor、Claude Code、Codex、Windsurf 全部能用。不需要为每个编辑器写独立插件。

完整技术栈一览

CLI 和 Web 用同一套核心逻辑，不同的运行时适配：

🖥

CLI 运行时

Node.js 原生运行。Tree-sitter 原生绑定。LadybugDB 原生。Worker 线程池并行。本地文件系统持久化。

🌐

Web 运行时

浏览器 + WASM。Tree-sitter WASM 解析。LadybugDB WASM 存储。Web Workers + Comlink 并行。内存会话级存储。

🎨

Web 前端

React 18 + TypeScript + Vite 构建。Tailwind v4 样式。Sigma.js + Graphology (WebGL) 图可视化。i18n 多语言支持。

🔗

AI 接口

CLI 通过 MCP（stdio）暴露工具。Web 通过 LangChain ReAct Agent 提供对话。Bridge 模式让 Web 连接 CLI 后端获得完整能力。

总结与练习

回顾 GitNexus 核心知识，通过综合练习巩固技能，探索代码图谱生态

核心知识回顾

🔒

零服务器理念

Tree-sitter WASM 解析代码、LadybugDB WASM 存储图谱、transformers.js 生成本地嵌入。三大 WASM 组件全部在浏览器运行，代码永远不离开你的机器。隐私、性能、零运维三赢。

⚡

WASM 三件套

Tree-sitter（AST 解析，14 种语言统一语义标签）+ LadybugDB（嵌入式图数据库，支持向量搜索）+ transformers.js（本地嵌入，WebGPU 加速）。CLI 用原生绑定，Web 用 WASM，共享同一套查询逻辑。

📈

12 阶段 DAG 管道

scan → structure → parse → crossFile → mro → communities → processes。解析用 Tree-sitter 提取符号，6 阶段调用解析推断类型，Leiden 算法聚类社区，执行流追踪入口到终点的完整调用链。

🌐

Sigma.js WebGL 可视化

Sigma.js + Graphology 渲染交互式网络图，WebGL GPU 加速支撑数千节点。Graph RAG Agent 基于 LangChain ReAct 提供自然语言问答，16 个 MCP 工具覆盖查询、影响分析、变更检测。

🎯

技术选型哲学

Tree-sitter 而非 LSP（轻量可 WASM 化）、LadybugDB 而非 Neo4j（嵌入式无服务器）、transformers.js 而非 OpenAI API（本地隐私）、MCP 而非自定义协议（一套服务多编辑器）。
