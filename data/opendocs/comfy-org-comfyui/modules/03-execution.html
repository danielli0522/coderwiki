数据在节点间流淌：执行引擎揭秘

从验证到执行，一条 prompt 走过的完整旅途

为什么需要执行引擎？

你画的节点图可能有几十个节点，连成复杂的网络。但计算机不能'同时'执行所有节点——就像地铁线路图上，列车必须按顺序从起点开到终点。

执行引擎的核心工作就是：确定顺序，按序执行，传递数据。

把执行引擎想象成一个快递分拣中心。每个包裹（数据）都有明确的目的地（下游节点），分拣员（引擎）要确保：

1
验证

包裹上的地址格式对不对？收件人存在吗？（类型检查、循环依赖检测）

2
排序

哪些包裹要先送？不能让收件人等不到上游的包裹。（拓扑排序）

3
执行

按排好的顺序逐个节点执行，把输出传给下游节点，直到所有输出节点完成

4
缓存

上次已经算过的结果如果输入没变，直接复用，不用重新算（智能缓存）

第一步：验证 Prompt

validate_prompt 函数是第一道关卡。它做什么？

CODE 

async def validate_prompt(prompt_id, prompt, partial_execution_list):
    outputs = set()
    for x in prompt:
        class_type = prompt[x]["class_type"]
        class_ = nodes.NODE_CLASS_MAPPINGS.get(class_type)
        if class_ is None:
            return (False, error, [], {})
        if hasattr(class_, "OUTPUT_NODE"):
            outputs.add(x)
    if len(outputs) == 0:
        return (False, error, [], {})
大白话

定义异步验证函数，接收 prompt ID 和节点字典...

准备一个集合来记录哪些节点是'输出节点'...

遍历 prompt 里的每一个节点...

取出这个节点的类型名称...

在全局注册表里查找这个类型——如果找不到...

说明自定义节点没安装，直接返回错误...

如果这个节点标记了 OUTPUT_NODE = True...

就把它记到输出集合里（SaveImage 就是输出节点）...

如果整个图连一个输出节点都没有...

返回错误：这个图没有终点，执行了也没意义。

⚠
常见报错

如果你看到 "Node 'XXX' not found. The custom node may not be installed"——这就是 validate_prompt 发现一个节点的 class_type 在 NODE_CLASS_MAPPINGS 里找不到。解决方案：安装对应的自定义节点。

智能缓存：不重复造轮子

如果一张图你生成两次，但只改了最后的 SaveImage 参数，前面的模型加载、文本编码都不需要重新算。ComfyUI 提供了多种缓存策略：

🔄

Classic 缓存

用完即丢——最保守的策略，每次执行都重新计算。适合调试场景。

📦

LRU 缓存

最近用过的留着，很久没用的丢掉。通过 --cache-lru N 设置大小。

💸

RAM Pressure 缓存

默认策略。根据可用内存动态决定保留多少缓存，内存紧张时自动清理。

CODE 

class CacheType(Enum):
    CLASSIC = 0
    LRU = 1
    NONE = 2
    RAM_PRESSURE = 3
大白话

用枚举定义四种缓存策略...

Classic：经典模式，用完就清...

LRU：保留最近使用的，丢弃最久没用的...

NONE：完全不缓存，每次都从头算...

RAM_PRESSURE：根据内存压力动态调整，默认选择。
