01

从第一个 Agent 任务说起

你用 AI 写过文案、回答过问题——但当你需要 AI 自己去"查资料再回答"时，它就不行了。这就是 Agent 要解决的问题。

这个项目在做什么？

想象你问 AI："华为最新手机是什么？它的主要卖点是什么？"

普通 AI 只能靠训练数据里已有的知识来猜。但如果它能自己上网搜索，拿到最新信息再回答你，那就完全不同了。

Hello-Agents 就是在教你构建这样的系统——一个能自主思考、调用工具、获取信息、做出决策的 智能体。

💡
核心理念：从"使用者"到"构建者"

这个项目的口号是让你从大语言模型的"使用者"蜕变为智能体系统的"构建者"。整本书围绕一个目标：让你亲手用代码搭建出能自主行动的 AI。

最早的"智能体"：ELIZA

智能体的历史可以追溯到 1966 年的 ELIZA——一个用 正则表达式 模拟心理治疗师的程序。它的核心思路是：匹配用户输入的句式，然后套用模板回复。

CODE 

# ELIZA 的规则库：模式 -> 回复模板
rules = {
    r'I need (.*)': [
        "Why do you need {0}?",
        "Would it really help you to get {0}?",
    ],
    r'I am (.*)': [
        "Did you come to me because you are {0}?",
        "How long have you been {0}?",
    ],
}
PLAIN CHINESE

定义一个字典（键值对集合），键是正则匹配模式，值是回复模板列表。

当用户说 "I need xxx" 时触发这条规则，{0} 会被替换为 xxx 的内容。

提供多个备选回复，让对话看起来不那么机械。

另一条规则：匹配 "I am xxx"，然后追问关于这个状态的问题。

ELIZA 的关键在于它不做真正的"理解"——只是匹配模式、填模板。

📖
知识卡片：从规则到智能

ELIZA 用固定的"如果用户说 X，就回 Y"规则。现代 Agent 用 LLM 来"理解"用户意图，然后自己决定该做什么。从"写死的规则"到"自主推理"，这是 60 年智能体进化最核心的变化。

智能体的心脏：思考-行动循环

Hello-Agents 项目中最核心的模式叫 ReAct（Reasoning + Acting）。它让 AI 交替地"思考"和"行动"，像一个人在解题：先想清楚下一步做什么，然后去做，再根据结果继续想。

智能体的核心角色

一个智能体系统就像一支乐队：有人负责思考，有人负责干活，有人负责记忆。认识这些角色是理解整个代码库的第一步。

代码库里的"角色表"

打开 Hello-Agents 的 code/chapter7/ 目录，你会看到几个核心文件。每个文件扮演一个角色：

🧠
my_llm.py — 大脑

封装了 OpenAI 兼容接口 的调用逻辑，负责把消息发给 LLM、拿回回复

🔧
my_calculator_tool.py — 工具

定义了 Agent 可以调用的工具（搜索、计算器等），是 Agent 的"手和脚"

🤖
my_react_agent.py — 决策者

实现 ReAct 循环：调用 LLM 思考 → 解析出要执行的工具 → 执行工具 → 把结果喂回 LLM → 循环

🎯
my_simple_agent.py — 基础版

更简单的 Agent 变体，支持纯对话和可选的工具调用，适合理解基本结构

大脑如何工作：LLM 客户端

所有 Agent 的"智能"都来自一个统一的大脑——HelloAgentsLLM 类。它封装了与 流式响应。

CODE 

class HelloAgentsLLM:
    def __init__(self, model, apiKey, baseUrl):
        self.model = model
        self.client = OpenAI(
            api_key=apiKey,
            base_url=baseUrl
        )

    def think(self, messages):
        response = self.client.chat.completions.create(
            model=self.model,
            messages=messages,
            stream=True,
        )
PLAIN CHINESE

定义一个 LLM 客户端类，管理"用哪个模型"和"怎么连接"。

构造函数接收模型名称、API 密钥和服务地址三个关键参数。

使用 OpenAI 库创建客户端实例——关键是 base_url 可以指向任何兼容服务。

think 方法是核心：把对话历史发给模型，获取回复。

stream=True 开启流式输出，让回复像打字机一样逐字返回。

03

消息与工具如何协作

Agent 不只是调用 LLM 那么简单——它要解析 LLM 的输出、识别工具调用请求、执行工具、再把结果喂回去。这是一条精密的数据流水线。

给 AI 的"工作手册"：Prompt 模板

Agent 不会"天生"知道怎么调用工具。我们通过一个精心设计的 Prompt 模板 来告诉 LLM："你有哪些工具可用，以及该怎么调用它们。"

CODE 

REACT_PROMPT_TEMPLATE = """
你是一个有能力调用外部工具的智能助手。
可用工具如下：
{tools}

请严格按照以下格式进行回应：
Thought: 你的思考过程
Action: tool_name[tool_input]
  或 Finish[最终答案]
"""
PLAIN CHINESE

这是一段多行字符串，作为发给 LLM 的系统指令。

先告诉 LLM 它的角色定位：一个能调用外部工具的助手。

{tools} 是占位符，运行时会被替换为实际可用的工具描述。

核心规则：要求 LLM 严格遵循 Thought/Action 的输出格式。

Action 有两种形式：调用工具 或 直接输出最终答案（Finish）。

💡
Prompt 工程 = 写合同

这个模板本质上是一份"合同"——你和 LLM 约定好了它的输出格式。只要它遵守格式，你的代码就能解析出"它想做什么"。如果格式乱了，整个系统就崩了。这就是为什么模板里的措辞要非常精确。

解析 AI 的"意图"

LLM 返回一段文本，代码需要从中提取出"思考了什么"和"要做什么"。这靠的是 正则表达式。

CODE 

def _parse_output(self, text: str):
    thought_match = re.search(
        r"Thought:\s*(.*?)(?=\nAction:|$)",
        text, re.DOTALL)
    action_match = re.search(
        r"Action:\s*(.*?)$",
        text, re.DOTALL)
    return thought, action
PLAIN CHINESE

接收 LLM 输出的完整文本，开始解析。

用正则找 "Thought:" 后面的内容，一直匹配到 "Action:" 或文本末尾。

re.DOTALL 让点号也能匹配换行符——因为思考内容可能跨行。

同样用正则找 "Action:" 后面的内容到文本末尾。

返回提取到的 thought（思考内容）和 action（行动指令）。

工具注册与执行

在 Agent 运行之前，你需要先"注册"可用的工具。就像给新员工配工位上的工具箱——他不认识的工具就不能用。

1
定义工具函数

写一个普通的 Python 函数，比如 search(query)，接收参数、返回结果

2
注册到 ToolExecutor

调用 registerTool("Search", description, search)，给它起个名字和描述

3
Agent 自动发现

Prompt 模板里的 {tools} 会被替换为所有已注册工具的描述，LLM 就知道有什么可用

4
按名字调用

LLM 输出 Action: Search[华为最新手机]，代码用名字查到对应函数并执行

📖
工具调用格式的演变

第 4 章用 tool_name[input] 这种自定义格式。第 7 章进化到 [TOOL_CALL:name:params] 标签。再到后来用 OpenAI 原生的 Function Calling。 越来越标准，越来越不容易出错。

如果 LLM 没有按照 "Thought: ... Action: ..." 格式输出，Agent 会发生什么？

04

多智能体协作模式

一个人做不了所有事。当任务变复杂，你需要多个 Agent 分工合作——就像公司里有不同部门各司其职。

三种经典的"思考方式"

Hello-Agents 教你实现三种智能体范式，每种解决不同类型的问题：

🔄

ReAct

边想边做。每一步都先思考再行动，观察结果后继续。像侦探破案：发现线索 → 顺着线索查 → 发现新线索 → 继续追查。

📋

Plan-and-Solve

先做计划再执行。把大任务拆成小步骤，然后逐步完成。像旅行攻略：先列清单（订票→找房→打包），再逐项执行。

🪞

Reflection

做完之后自我检查。生成一个答案，然后让 AI 审视自己的答案，找出问题并改进。像学生写完作文后自己检查一遍。

ReAct 循环的核心代码

ReAct Agent 的核心是一个 while 循环——每一步都把 Prompt 发给 LLM，解析输出，执行工具，把观察结果追加到历史里，再循环。

CODE 

def run(self, question: str):
    current_step = 0
    while current_step < self.max_steps:
        prompt = TEMPLATE.format(
            tools=tools_desc,
            question=question,
            history=history_str)
        response = llm.think(messages)
        thought, action = _parse(response)
        if action.startswith("Finish"):
            return final_answer
        observation = tool_func(tool_input)
        history.append(f"Observation: {observation}")
PLAIN CHINESE

Agent 的运行方法，接收用户的问题作为输入。

用一个计数器跟踪当前执行到第几步，防止无限循环。

把工具描述、问题、历史拼装成完整的 Prompt 发给 LLM。

从 LLM 的回复中解析出 thought 和 action 两个字段。

如果 LLM 觉得信息够了（action 以 Finish 开头），直接返回最终答案。

否则执行工具，把观察结果追加到对话历史中，下一轮继续。

05

评测、记忆与安全

让 Agent "能做事"只是第一步。真正的挑战是：怎么确保它做得好？怎么让它记住过去的经验？怎么防止它被恶意利用？

给 Agent 装上"记忆"

没有记忆的 Agent 就像金鱼——每次对话都从零开始。Hello-Agents 第 8 章深入讲解了 记忆系统：

📝

短期记忆

就是当前对话的上下文历史。每次调用 LLM 都带上之前的消息列表。用 self._history 存储在代码里。

🗄️

长期记忆

跨对话的知识持久化。用 MemoryTool 把重要信息存储到数据库，下次对话时可以检索。赛博小镇里的 NPC 就靠这个记住"昨天和张三聊了什么"。

🔍

RAG 检索

RAG 让 Agent 能从海量文档中精准找到需要的信息，而不是把所有内容都塞进上下文。

怎么知道 Agent 做得好不好？

第 12 章讲了一个关键问题： 智能体评估。 项目引入了 BFCL 和 GAIA 两个评估基准。

CODE 

# BFCL 评估：测试 function calling 能力
from bfcl import evaluate_agent

result = evaluate_agent(
    agent=my_agent,
    test_cases=[
        {"input": "北京的天气",
         "expected_tool": "get_weather",
         "expected_args": {"city": "北京"}},
    ]
)
PLAIN CHINESE

BFCL 专门测试 Agent 的 function calling 能力——它是否选对了工具。

每个测试用例定义了输入、期望调用的工具名和期望的参数。

评估器会对比 Agent 实际调用的工具和期望工具是否一致。

参数也要匹配——Agent 不能调对工具但传错参数。

⚠️
常见陷阱：只测正确率

很多人只看"Agent 回答对了几个问题"。但真正重要的是：它在什么情况下会出错？ 是工具选择错了？参数传错了？还是根本没意识到需要用工具？分类统计错误类型，才能有针对性地改进。

上下文工程：喂什么给 AI 很关键

第 9 章提出了一个重要概念：上下文工程——精心设计每次发给 LLM 的信息，让它能做出最好的决策。

1
选什么信息放进 Prompt

不是所有历史消息都有用。要选择最相关的事实、最近的对话、最适合的工具描述

2
控制信息的顺序和格式

把重要信息放在 Prompt 开头或结尾（LLM 对这些位置更敏感），用清晰的分隔符区分不同部分

3
管理 Token 预算

LLM 有 Token 上限。上下文越长，成本越高，延迟越大。要像理财一样管理你的 token 预算

你的 Agent 需要回答关于公司内部文档的问题。哪种方法最靠谱？

从源码中提取的硬限制

max_steps = 5 ReAct Agent 默认最大循环步数。防止无限循环，但复杂任务可能不够用（可在初始化时修改）
max_tool_iterations = 3 SimpleAgent 的工具调用最大轮次。每次调用 LLM → 检查工具调用 → 执行 → 喂回结果，最多 3 轮
temperature = 0 默认温度为 0（最确定性的输出）。Agent 需要可预测的行为，不适合高创造性场景
TOOL_CALL 正则 工具调用解析依赖正则匹配，格式不严格就会解析失败。这是纯文本解析方案的固有局限
