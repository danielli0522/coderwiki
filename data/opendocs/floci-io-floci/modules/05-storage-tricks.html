Making It Last: Storage & Smart Tricks

A photographer's workflow — polaroids that fade by morning, prints stored in a box, digital photos that auto-sync, and film negatives developed immediately in a fireproof safe. Four ways to remember, each with a different trade-off.

Four Ways to Remember

Floci has 4 storage modes, controlled by the FLOCI_STORAGE_MODE env var. Each one balances speed and safety differently.

🎨
Memory (default)

Entirely in RAM. Data lost on stop. Like polaroids that fade by morning.

Best for: CI pipelines, quick tests.

📦
Persistent

Loaded at startup, flushed on graceful shutdown. Like prints stored in a box.

Best for: local dev with state preservation.

☁️
Hybrid

In-memory speed + async flush every 5s. Like digital photos that auto-sync.

Best for: best balance of speed and safety.

🔥
WAL

Write-Ahead Log — every mutation logged to disk before responding. Like film negatives in a fireproof safe.

Best for: maximum durability.

FLOCI_STORAGE_MODE memory | persistent | hybrid | wal
FLOCI_STORAGE_PERSISTENT_PATH where data lives on disk (default: ./data)

Multi-Account Isolation

If your AWS access key ID is exactly 12 digits, Floci uses it as the account ID. Resources from one account are invisible to another — full isolation, zero config.

bash # Account 1111 creates a queue AWS_ACCESS_KEY_ID=111111111111 \ aws sqs create-queue \ --queue-name orders # Account 2222 creates a queue AWS_ACCESS_KEY_ID=222222222222 \ aws sqs create-queue \ --queue-name orders
In plain English
Account 1111 creates a queue called orders.
Account 2222 creates a queue also called orders.
These are TWO SEPARATE queues — full isolation!
Each account sees only its own resources. Same names, different worlds.
💡
Why this matters

You can run multiple test suites in parallel, each with its own account ID, and they'll never interfere with each other. No need for separate containers or complex namespacing.

LocalStack Drop-In Replacement

Already using LocalStack? Floci is a drop-in replacement — same port (4566), same credentials, same SDK calls. Just swap the image name.

LOCALSTACK_HOST → FLOCI_HOSTNAME
PERSISTENCE=1 → FLOCI_STORAGE_MODE=persistent
DEBUG=1 → QUARKUS_LOG_LEVEL=DEBUG

Init scripts mounted at /etc/localstack/init/ run unchanged — your existing setup scripts keep working without modification.

📚
Good to know

The -compat image variant includes Python 3, AWS CLI, and boto3 pre-configured. Use it if your init scripts call the AWS CLI.

The Big Picture

Let's zoom out and see everything you've learned, organized as the architecture it really is:

floci/
HTTP Router (JAX-RS / Vert.x) — routes all incoming AWS API calls
Stateless Services — SSM, SQS, SNS, IAM, STS, KMS, EventBridge...
Stateful Services — S3, DynamoDB, DynamoDB Streams
Container Services — Lambda, RDS, ElastiCache, ECS, EC2, EKS...
Storage Backend — memory | persistent | hybrid | wal
💡
Aha!

Floci is written in Java and compiled to a native binary using GraalVM. This is why it starts in 24ms, uses only 13 MiB of memory, and ships as a ~40 MB native executable. No JVM, no runtime overhead — just a lean, fast AWS emulator.

Scenario

Your team runs integration tests in CI. Tests need a clean AWS environment each run, and they must finish fast. Which storage mode and image tag should you use?

Why each answer matters

Option A (WAL + compat) — WAL adds disk I/O overhead and the compat image is larger. You do not need persistence in CI — each test gets a fresh instance.

Option C (Hybrid + nightly) — Hybrid mode adds unnecessary overhead in CI. And nightly images may include unstable changes that break your tests unpredictably.
