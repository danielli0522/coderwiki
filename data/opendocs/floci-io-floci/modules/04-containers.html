The Real Stuff: Docker Containers

A flight simulator with real engines — most things are simulated (the clouds, the weather), but the engines are real. If your plane flies in the simulator, it will fly in real life too.

Why Real Containers Matter

Most AWS emulators take a shortcut: they mock everything. Ask them to save data? "Sure, saved!" — but nothing actually happened. Run a SQL query? "Here's a fake result!" — your typo goes undetected.

Floci takes a different approach. It simulates the AWS control plane (the APIs you call), but for the heavy lifting, it runs real engines:

🗄️

Real PostgreSQL

Spins up an actual postgres:16-alpine container. Your SQL queries execute against a real database engine — syntax errors, constraint violations, all genuine.

⚡

Real Redis

Runs valkey/valkey:8 (Redis-compatible). Every SET, GET, and PUBSUB command works exactly as it would in production.

🐍

Real Python / Node.js / Java

Lambda functions execute in real runtimes using official AWS ECR images.

💡
Aha!

The difference matters for debugging. If your SQL query has a syntax error, a mock will never catch it. A real PostgreSQL container will tell you exactly what is wrong — the same error you would see in production.

🗄️
RDS

⚡

📨

🖥️
EC2

⎈
ECS & EKS

🔨
CodeBuild

Executes real buildspec phases in Docker containers. Install, pre_build, build, post_build — each phase runs for real.

Lambda — The Star of the Show

Lambda has the most sophisticated container management in Floci. Each runtime maps to a real AWS image:

CODE 

Runtime         → Docker Image
python3.12      → public.ecr.aws/lambda/python:3.12
nodejs20.x      → public.ecr.aws/lambda/nodejs:20.x
java21          → public.ecr.aws/lambda/java:21
go1.x           → public.ecr.aws/lambda/go:1
provided.al2023 → public.ecr.aws/lambda/provided:al2023
PLAIN ENGLISH

Each Lambda runtime maps to a real AWS container image from ECR.

Python 3.12 uses the exact same image AWS runs in production.

Node.js 20 — same image, same Node version, same modules.

Java 21, Go 1.x — all production-grade runtime images.

Even custom runtimes (provided.al2023) use the real Amazon Linux 2023 base.

Your code runs in the exact same environment locally as it would in AWS. Same libraries, same file paths, same environment variables.

Warm Pool Containers stay alive after first invoke — no cold start on subsequent calls
Function URLs Direct HTTP endpoints for Lambda functions — invoke via curl or browser
Event Source Mapping Lambda can be triggered by SQS, Kinesis, or DynamoDB Streams — real event-driven architecture
