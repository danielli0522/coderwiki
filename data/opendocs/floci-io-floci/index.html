What Happens When You Type docker compose up

A rehearsal studio for musicians — you get a fully equipped stage with every instrument (all AWS services) to practice on before the real concert (production AWS). No tickets needed, no security guards checking IDs.

Meet Floci

Floci is a local emulator that runs all of AWS on your machine. No AWS account needed, no credit card, no internet required.

The name comes from floccus — the cloud formation that looks like popcorn. Light, fluffy, everywhere you need it. That's exactly what Floci is: your own personal cloud, wherever you are.

1
Start the container

Run docker compose up in your terminal. One command, zero configuration.

2
Point your SDK to localhost:4566

That's the local port Floci listens on. Same API, same protocol, different address.

3
Develop freely

Create buckets, write items, invoke functions — everything works as if it were real AWS.

docker-compose.yml services: floci: image: floci/floci:latest ports: - "4566:4566" volumes: - ./data:/app/data
In plain English
Define a service called floci
Pull the latest Floci image from Docker Hub
Expose port 4566 so your code can reach Floci at localhost
Mount a local volume so your data survives restarts

The Lecture Hall — Stateless Services

These services handle requests but don't persist data between calls. Like a lecture hall — the professor answers your question, but doesn't write it down for next time.

📬
SQS & SNS

Message queues and pub/sub notifications. SQS queues messages for reliable delivery; SNS broadcasts events to multiple subscribers. Like a campus mailroom — drop a letter, and it gets routed to the right mailbox.

🔑
IAM & STS

Identity and access management. IAM defines who can do what (policies, roles, users). STS issues temporary credentials. Like student ID cards — each request proves who you are, independently of any previous request.

📡
EventBridge & Step Functions

Event routing and workflow orchestration. EventBridge connects events to targets (like a campus events board). Step Functions chains multiple service calls into coordinated workflows (like a graduation checklist).

The Library — Stateful Services

These services remember data between requests. Like a library — the books stay on the shelves even after you leave.

📁
S3 — Object Storage

Stores files, images, and backups as objects in buckets. Supports versioning (keep multiple versions of a file), multipart uploads (upload large files in chunks), and pre-signed URLs (temporary access links).

📊
DynamoDB — NoSQL Database

Fast key-value lookups, queries, and scans. Supports secondary indexes (query by different attributes), TTL (automatically expire old items), and transactions (multi-item atomic operations).

📼
DynamoDB Streams — Change Data Capture

Watches for item-level changes in DynamoDB tables and triggers Lambda functions in response. If a record is added, updated, or deleted — Streams catches it and kicks off the next step.

The Makerspace — Container Services 🐳

These services spin up real Docker containers. When you invoke a Lambda function, Floci pulls a real container image from ECR and runs your code inside it.

The AWS Wire Protocol

All AWS services speak the same "language" — they communicate using a standard wire protocol (JSON or XML over HTTPS). This means your SDK already knows exactly how to format requests. You just need to tell it where to send them.

Java 

DynamoDbClient client = DynamoDbClient.builder()
    .endpointOverride(
        URI.create("http://localhost:4566"))
    .region(Region.US_EAST_1)
    .credentialsProvider(
        StaticCredentialsProvider.create(
            AwsBasicCredentials.create(
                "test", "test")))
    .build();
PLAIN ENGLISH

Create a DynamoDB client — your doorway to the database service.

Tell it to talk to Floci instead of real AWS by overriding the endpoint.

Pick any region — Floci does not care which one you choose.

Use dummy credentials — "test/test" works just fine.

Done! This client now sends every request to Floci instead of AWS.

Authentication — SigV4

Floci validates request signatures using SigV4, the same protocol as real AWS. This means your authentication code works identically in development and production — no special cases, no conditional logic.

1
Your SDK signs every request automatically

When you call any AWS API method, the SDK computes a SigV4 signature from your credentials, the current timestamp, and the request body. You never do this manually.

2
Floci checks the signature

Not because Floci needs to verify your identity, but because your SDK will send it anyway. By validating the seal, Floci ensures your production auth code gets tested locally.

3
Any credentials work locally

Use "test" / "test" or any string you like. Floci accepts all validly-signed requests, so you are not locked into managing real AWS keys during development.

💡
Aha!

SigV4 is like a wax seal on a letter. AWS uses it to verify that the request actually came from you and was not tampered with. Floci checks the seal too — not because it needs to verify your identity, but because your SDK will send it anyway. This way, your production auth code gets tested locally.

The Real Stuff: Docker Containers

A flight simulator with real engines — most things are simulated (the clouds, the weather), but the engines are real. If your plane flies in the simulator, it will fly in real life too.

Why Real Containers Matter

Most AWS emulators take a shortcut: they mock everything. Ask them to save data? "Sure, saved!" — but nothing actually happened. Run a SQL query? "Here's a fake result!" — your typo goes undetected.

Floci takes a different approach. It simulates the AWS control plane (the APIs you call), but for the heavy lifting, it runs real engines:

🗄️

Real PostgreSQL

Spins up an actual postgres:16-alpine container. Your SQL queries execute against a real database engine — syntax errors, constraint violations, all genuine.

⚡

Real Redis

Runs valkey/valkey:8 (Redis-compatible). Every SET, GET, and PUBSUB command works exactly as it would in production.

🐍

Real Python / Node.js / Java

Lambda functions execute in real runtimes using official AWS ECR images.

💡
Aha!

The difference matters for debugging. If your SQL query has a syntax error, a mock will never catch it. A real PostgreSQL container will tell you exactly what is wrong — the same error you would see in production.

Making It Last: Storage & Smart Tricks

A photographer's workflow — polaroids that fade by morning, prints stored in a box, digital photos that auto-sync, and film negatives developed immediately in a fireproof safe. Four ways to remember, each with a different trade-off.

Four Ways to Remember

Floci has 4 storage modes, controlled by the FLOCI_STORAGE_MODE env var. Each one balances speed and safety differently.

🎨
Memory (default)

Entirely in RAM. Data lost on stop. Like polaroids that fade by morning.

Best for: CI pipelines, quick tests.

📦
Persistent

Loaded at startup, flushed on graceful shutdown. Like prints stored in a box.

Best for: local dev with state preservation.

☁️
Hybrid

In-memory speed + async flush every 5s. Like digital photos that auto-sync.

Best for: best balance of speed and safety.

🔥
WAL

Write-Ahead Log — every mutation logged to disk before responding. Like film negatives in a fireproof safe.

Best for: maximum durability.

FLOCI_STORAGE_MODE memory | persistent | hybrid | wal
FLOCI_STORAGE_PERSISTENT_PATH where data lives on disk (default: ./data)

Multi-Account Isolation

If your AWS access key ID is exactly 12 digits, Floci uses it as the account ID. Resources from one account are invisible to another — full isolation, zero config.

bash # Account 1111 creates a queue AWS_ACCESS_KEY_ID=111111111111 \ aws sqs create-queue \ --queue-name orders # Account 2222 creates a queue AWS_ACCESS_KEY_ID=222222222222 \ aws sqs create-queue \ --queue-name orders
In plain English
Account 1111 creates a queue called orders.
Account 2222 creates a queue also called orders.
These are TWO SEPARATE queues — full isolation!
Each account sees only its own resources. Same names, different worlds.
💡
Why this matters

You can run multiple test suites in parallel, each with its own account ID, and they'll never interfere with each other. No need for separate containers or complex namespacing.

LocalStack Drop-In Replacement

Already using LocalStack? Floci is a drop-in replacement — same port (4566), same credentials, same SDK calls. Just swap the image name.

LOCALSTACK_HOST → FLOCI_HOSTNAME
PERSISTENCE=1 → FLOCI_STORAGE_MODE=persistent
DEBUG=1 → QUARKUS_LOG_LEVEL=DEBUG

Init scripts mounted at /etc/localstack/init/ run unchanged — your existing setup scripts keep working without modification.

📚
Good to know

The -compat image variant includes Python 3, AWS CLI, and boto3 pre-configured. Use it if your init scripts call the AWS CLI.
