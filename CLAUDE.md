- # 🔄 开发工作流程

## 开发哲学
- 增量优于全部重构：通过小步迭代的方式持续改进代码
- 代码要清晰而非聪明：优先考虑代码的可读性和维护性
- 极致简洁：质疑任何非必要的抽象层、接口或间接调用

## 标准工作流
1. 规划 - 明确需求和目标
2. 写测试 - 先写测试用例  
3. 实现 - 编写功能代码
4. 重构 - 优化代码结构
5. 提交 - 提交代码变更

## 问题解决预案
**"卡住怎么办"预案**：
1. 尝试3次失败后，必须停下来
2. 记录失败原因和过程
3. 研究替代方案
4. 反思根本问题

## 决策框架
当有多种方案时，按以下优先级选择：
1. 可测试性 - 代码是否容易测试
2. 可读性 - 代码是否容易理解
3. 一致性 - 是否符合现有代码风格
4. 简单性 - 实现是否简洁

## YAGNI 原则
- 始终选择能满足当前需求的最简方案
- 除非业务复杂性明确要求，否则**禁止**主动引入：
  - CQRS
  - 事件溯源
  - 状态机
  - 消息队列等复杂模式

## 代码架构维护性
- 单文件不要超过 600 行
- 优先选择具体实现
- 仅在出现**明确的、重复的模式**或**需要替换实现**时才进行抽象

## 工作流
- 完成一系列代码更改后务必进行类型检查
- 为了性能考虑，优先运行单个测试而非整个测试套件
